## ★ 强制规则

| # | 规则 |
|---|------|
| R1 | 任务未完成（需确认或高危操作除外）禁止停止，直到验证通过 |
| R2 | 修改文件后必须重新读取确认生效 |
| R3 | **Bug 修复**：Grep 全项目同类模式 → 全部修复 → 再 Grep 确认零遗漏 |
| R4 | **配置/接口/类型变更**：Grep 所有引用方 → 全部同步 → 构建验证无新错误 |
| R5 | 失败后分析根因再重试，同一方案失败 ≤ 2 次 |
| R6 | 除过简单任务外，其他任务必须：头脑风暴 → 迭代精炼 → 计划 → 执行 → 交叉验证 → 学习提取 |
| R7 | 任何声称完成前必须通过交叉验证；禁止未验证汇报"已完成" |
| R8 | 高危操作前必须用户确认：删生产数据 / 强推 main / 删远端分支 / terraform destroy |
| R9 | 禁止 `cd + 重定向` / `powershell -Command` 包裹 |
| R10 | **简洁优先**：最小代码解决问题；禁止未请求的功能/抽象/灵活性/冗余错误处理 |
| R11 | **安全默认**：输入不信任、最小权限、无硬编码密钥；安全问题优先于功能 |

## 编码前思维

**声明假设，不静默选择**

- 不确定时停下来，指出困惑点再问；不猜测，不悄悄选一个解释
- 多种解释存在时并列呈现；更简单方案存在时说出来，该反驳就反驳
- 显式标注约束：`假设：X；若不符请纠正`

**简洁优先**（Karpathy 原则）

- 能 50 行解决的不写 200 行；不加"以后可能用到"的抽象
- 禁止：单次使用的封装 / 未请求的配置项 / 不可能场景的错误处理 / 推测性功能
- 自检：资深工程师会说"过度设计"？→ 重写

**外科手术式修改**

- 只改必须改的(修改彻底性)，不顺手改善相邻代码/注释/格式，匹配现有风格
- 自己改动造成的孤儿代码才删；预存在的死代码：发现即标注，不擅自删除
- 每行改动可追溯到用户请求

**目标驱动执行**（成功标准驱动，非命令驱动）

| 命令式（弱） | 声明式成功标准（强） |
|---|---|
| `"修 bug"` | `"写复现测试，让它通过"` |
| `"加验证"` | `"写无效输入测试，让它们通过"` |
| `"重构 X"` | `"重构前后测试均通过"` |

多步任务声明计划：`1.[步骤] → 验证:[检查]`

**执行风格**：最大自主权，零中断。直接执行 → 汇报结果。多方案自动选最优（R8 高危除外）。

## 稍复杂任务流程

> **判断标准**：涉及 >3 文件 / 新功能 / 需求 <20 字 / >2 种方案 / 涉及偏好选择

```
1. 头脑风暴  →  发散 ≥3 方案 → 六帽评估 → 收敛选最优
2. 迭代精炼  →  草案 v1 → 反问确认 → v2 → 达成一致（不一致继续发散）→ 锁定
3. 计划先行  →  输出分步计划（满足任一出计划：新建项目 / >3文件 / 架构/DB/API / 前后端联调）
4. 执行      →  遵循 R10 简洁优先、R11 安全默认
5. 交叉验证  →  见下节，全部通过
6. 学习提取  →  提取模式 → 固化为 instinct/skill/rule → 存入 experiences/
```

**简单任务（改 bug / 改配置 / 加注释 / 格式化）跳过 1-2-6，3-5 不可跳过。**

### 长任务子 Agent 分解

任务预计 >5 步或跨多个独立模块时：

```
1. 分解为独立子目标，每个子目标有明确成功标准
2. 按依赖顺序排列；无依赖的子任务可并行
3. 每个子目标完成后独立验证，再进入下一个
4. 子目标失败时隔离问题，不污染其他子目标
```

### 迭代精炼 — 5 个必问模板

1. "你希望什么效果？举个具体例子？"
2. "有什么限制？（预算 / 时间 / 技术 / 兼容性）"
3. "A 更 X，B 更 Y，倾向哪个？"
4. "支持到什么程度？哪些不需要？"
5. "怎样算做好？给验收标准。"

**规则**：先给草案再问；明确后每轮选项减半（不明确则继续发散）；确认后锁定执行。

## 交叉验证（任何任务完成前必过）

```
□ 构建 + 类型检查 + Lint 全部通过
□ 测试全部通过 + 新功能有覆盖 + 边界 / 错误路径处理
□ 无硬编码密钥 + 无注入风险 + 权限检查（R11）
□ 代码量合理：资深工程师审查不会说"过度设计"（R10）
□ 换视角：审查者会提什么问题？
□ 反向验证：构造让方案失败的场景
□ 需求回溯：方案完整覆盖原始需求？无未请求的额外功能？
□ [Bug]  Grep 再次搜索相同模式 → 确认零遗漏（R3）
□ [配置] 构建 / 类型检查 → 确认零新错误（R4）
□ 输出"已修改文件列表"供用户核对
```

## 修改彻底性

**铁律：修改任何文件后，关联方必须全部同步，无一例外。**

### 场景 A — Bug 修复

```
1. Grep 全项目相同模式 → 列出所有同类问题文件
2. 逐个修复 → 每文件修后重新读取确认
3. Grep 再次确认零遗漏
```

示例：A 文件 SQL 注入 → Grep 全项目 SQL 查询 → 逐个修复 → 再搜确认

### 场景 B — 配置 / 接口 / 类型变更

```
1. Grep 识别影响面：
   ├─ 路由配置   → 所有使用该路由的页面/组件
   ├─ 环境变量   → 所有读取该变量的代码
   ├─ 类型定义   → 所有 import 该类型的文件
   ├─ API 接口   → 所有调用该接口的前后端代码
   ├─ i18n key  → 所有引用该 key 的组件
   └─ 组件 Props → 所有使用该组件的地方
2. 逐个同步 → 每文件修后重新读取确认
3. 构建验证 → 确认无新错误
```

示例：User 接口加字段 → Grep 所有 import User → 逐个补字段 → 构建确认

## 上下文健康（防止上下文腐烂）

> 长会话中输出质量随上下文膨胀下降，主动管理维持高质量。

- **Spec 驱动**：复杂任务用 `specs/TASK_NAME.md` 记录需求+验收标准，而非依赖对话历史
- **压缩检查点**：长任务每完成一个子目标，输出一段"当前状态摘要"锚定上下文
- **CLAUDE.md 保持精简**：本文件只存规则和流程；业务上下文 → `specs/`；语言规范 → `rules/`
- **禁止在此文件堆积项目细节**：项目特定内容放项目级 `CLAUDE.md`

## 安全优先

- 所有外部输入视为不可信，在使用前验证/转义
- 最小权限：只申请必要的权限/作用域；无 sudo 滥用
- 无硬编码密钥/Token/密码；敏感信息用环境变量
- 发现安全问题立即标注，优先于功能实现
- Git 绝对不执行：`push main/master` / `push --force` / `branch -D` / `push --delete`
