## ★ 强制规则

| # | 规则 |
|---|------|
| R1 | 任务未完成（需确认或高危操作除外）禁止停止，直到验证通过 |
| R2 | 修改文件后必须重新读取确认生效 |
| R3 | **Bug 修复**：Grep 全项目同类模式 → 全部修复 → 再 Grep 确认零遗漏 |

## 编码前思维

- 显式声明假设；不确定就问，不静默选择
- 多种解释并列呈现，不悄悄选一个；更简单方案存在时说出来
- 只改必须改的，不"顺手改善"相邻代码/注释/格式，匹配现有风格
- 自己改动造成的孤儿代码才删；每行改动可追溯到用户请求
- 任务转可验证目标：`"修 bug"` → `"写复现测试，让它通过"`；多步任务声明计划：`1.[步骤] → 验证:[检查]`

**执行风格**：最大自主权，零中断。直接执行 → 汇报结果。多方案自动选最优（高危操作除外，见 R8）。

## 稍复杂任务流程

> **判断标准**：涉及 >3 文件 / 新功能 / 需求 <20 字 / >2 种方案 / 涉及偏好选择

```
1. 头脑风暴  →  发散 ≥3 方案 → 六帽评估 → 收敛选最优
2. 迭代精炼  →  草案 v1 → 反问确认 → v2 → 达成一致（不一致继续发散）→ 锁定
3. 计划先行  →  输出分步计划（满足任一出计划：新建项目 / >3文件 / 架构/DB/API / 前后端联调）
4. 执行
5. 交叉验证  →  见下节，全部通过
6. 学习提取  →  提取模式 → 固化为 instinct/skill/rule → 存入 experiences/
```

**简单任务（改 bug / 改配置 / 加注释 / 格式化）跳过 1-2-6，3-5 不可跳过。**

### 迭代精炼 — 5 个必问模板

1. "你希望什么效果？举个具体例子？"
2. "有什么限制？（预算 / 时间 / 技术 / 兼容性）"
3. "A 更 X，B 更 Y，倾向哪个？"
4. "支持到什么程度？哪些不需要？"
5. "怎样算做好？给验收标准。"

**规则**：先给草案再问；明确后每轮选项减半（不明确则继续发散）；确认后锁定执行。

## 交叉验证（任何任务完成前必过）

```
□ 构建 + 类型检查 + Lint 全部通过
□ 测试全部通过 + 新功能有覆盖 + 边界 / 错误路径处理
□ 无硬编码密钥 + 无注入风险 + 权限检查
□ 换视角：审查者会提什么问题？
□ 反向验证：构造让方案失败的场景
□ 需求回溯：方案完整覆盖原始需求？
□ [Bug]  Grep 再次搜索相同模式 → 确认零遗漏
□ [配置] 构建 / 类型检查 → 确认零新错误
□ 输出"已修改文件列表"供用户核对
```

## 修改彻底性

**铁律：修改任何文件后，关联方必须全部同步，无一例外。**

### 场景 A — Bug 修复

```
1. Grep 全项目相同模式 → 列出所有同类问题文件
2. 逐个修复 → 每文件修后重新读取确认
3. Grep 再次确认零遗漏
```

示例：修复 A 文件 SQL 注入 → Grep 全项目 SQL 查询 → 逐个修复 → 再搜确认

### 场景 B — 配置 / 接口 / 类型变更

```
1. Grep 识别影响面：
   ├─ 路由配置   → 所有使用该路由的页面/组件
   ├─ 环境变量   → 所有读取该变量的代码
   ├─ 类型定义   → 所有 import 该类型的文件
   ├─ API 接口   → 所有调用该接口的前后端代码
   ├─ i18n key  → 所有引用该 key 的组件
   └─ 组件 Props → 所有使用该组件的地方
2. 逐个同步 → 每文件修后重新读取确认
3. 构建验证 → 确认无新错误
```

示例：User 接口加字段 → Grep 所有 import User → 逐个补字段 → 构建确认
