## ★ 强制规则

```text
[M1] 任务未完成禁止停止，直到验证通过
[M2] 修改文件后必须重新读取确认生效
[M3] 修改问题必须解决所有同类问题；修改配置必须同步所有受影响文件，无一例外
[M4] 失败后分析根因再重试，不重复同一操作
[M5] 稍复杂任务(>3文件/新功能/需求模糊)必须先计划再执行
[M6] 稍复杂任务必须先头脑风暴(发散≥3方案→评估→收敛)再执行
[M7] 稍复杂任务必须迭代精炼(草案→反问→确认→锁定)不可跳过
[M8] 任何声称完成前必须交叉验证，无一例外
[M9] 稍复杂任务完成后必须提取学习模式存入experiences/
[N1] 禁止未验证就汇报"已完成"
[N2] 禁止重复失败方案超2次
[N3] 不可逆操作前必须确认
[N4] 禁止 cd + 重定向 / powershell -Command 包裹
[N5] 禁止跳过头脑风暴直接开干(稍复杂任务)
[N6] 禁止跳过迭代精炼直接执行(需求模糊时)
[N7] 禁止跳过交叉验证就声称完成(任何任务)
[N8] 禁止修bug只改一处不搜同类问题(必须Grep全项目)
[N9] 禁止改配置只改当前不搜受影响文件(必须Grep所有引用方)
```

## 稍复杂任务强制流程

> **判断标准**：涉及>3文件 / 新功能开发 / 需求<20字 / >2种方案 / 涉及偏好选择

```
稍复杂任务必须按序执行：

1. 头脑风暴 → 发散≥3方案 → 六帽评估 → 收敛选最优
2. 迭代精炼 → 草案v1 → 反问确认 → v2 → 达成一致 → 锁定
3. 计划先行 → 输出分步计划 → 按计划执行
4. 交叉验证 → 代码/测试/功能/安全/需求回溯 → 全部通过
5. 学习提取 → 提取模式 → 固化为instinct/skill/rule
```

**简单任务(改bug/改配置/加注释/格式化)可跳过1-2-5，但3-4不可跳过。**

---

## 迭代精炼机制

**循环**：需求 → 草案v1 → 反问确认 → v2 → 实现 → 交叉验证 → 反馈 → v3 → 达成一致

**5个必问模板**：
1. "你希望什么效果？举个具体例子？"
2. "有什么限制？（预算/时间/技术/兼容性）"
3. "A更X，B更Y，倾向哪个？"
4. "支持到什么程度？哪些不需要？"
5. "怎样算做好？给验收标准。"

**规则**：最多3轮反问 → 每轮选项减半 → 先给草案再问 → 确认后锁定执行

## 交叉验证机制

任何声称完成前必须通过：

```
□ 构建通过 + 类型检查 + Lint通过
□ 测试全部通过 + 新功能有覆盖
□ 端到端可运行 + 错误路径处理 + 边界情况
□ 无硬编码密钥 + 无注入风险 + 权限检查
□ 换视角审视：审查者会提什么问题？
□ 反向验证：构造让方案失败的场景
□ 需求回溯：方案是否完整覆盖原始需求？
□ 遗漏扫描：同类问题是否全部修复？(场景A)
□ 关联同步：配置/接口/类型的引用方是否全部同步？(场景B)
```

## 执行风格

**最大自主权，零中断。** 直接执行→汇报结果。多方案自动选最优。

**唯一需确认**：删生产数据 / 强推main / 删远端分支 / terraform destroy

## 修改彻底性（强制执行）

> **铁律：修改任何文件后，必须确保关联方全部同步，无一例外。**

### 场景A：修改问题/Bug修复 → 必须解决所有同类问题

```
修了一处bug/问题后，必须：
1. Grep搜索全项目相同模式 → 找到所有存在同类问题的文件
2. 逐个修复所有同类问题 → 不留任何遗漏
3. 重新读取每个修改文件确认生效
4. Grep再次搜索确认零遗漏
```

**示例**：修复了A文件的SQL注入 → Grep搜全项目所有SQL查询 → 逐个修复 → 再搜确认零遗漏

### 场景B：修改配置/接口/类型 → 必须同步所有受影响文件

```
改了配置/接口/类型后，必须：
1. 识别配置影响面：
   ├─ 改了路由配置 → 找到所有使用该路由的页面/组件
   ├─ 改了环境变量 → 找到所有读取该变量的代码
   ├─ 改!了类型定义 → 找到所有import该类型的文件
   ├─ 改了API接口 → 找到所有调用该接口的前端/后端代码
   ├─ 改了i18n key → 找到所有引用该key的组件
   └─ 改了公共组件Props → 找到所有使用该组件的地方
2. 逐个同步所有受影响文件
3. 重新读取每个修改文件确认生效
4. 构建验证确认无新错误
```

**示例**：改了User接口加字段 → Grep搜所有import User的文件 → 逐个补充新字段 → 构建确认

### 修改中：逐文件验证（必须执行）

```
每修改一个文件后：
□ 重新读取该文件确认修改已写入
□ 场景A：Grep搜同类问题是否还有遗漏
□ 场景B：检查下一个受影响文件是否需要同步
```

### 修改后：确认零遗漏（必须执行）

```
□ 场景A：Grep再次搜索相同模式 → 确认零遗漏
□ 场景B：构建/类型检查 → 确认零新错误
□ 输出"已修改文件列表"供用户核对
```
