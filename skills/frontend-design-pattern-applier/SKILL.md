# Frontend Design Pattern Applier

## 目标

识别前端代码中的僵化、高耦合的面向过程逻辑,通过智能分析推荐最合适的设计模式,并辅助用户完成重构。

## 约束条件

- **输入限制**: 仅支持单个模块或最多 5 个文件的分析,禁止整个项目级别分析
- **语言支持**: TypeScript/JavaScript 前端代码
- **重构阈值**: 代码复杂度达到一定水平才推荐模式重构,避免过度设计

## **反过度设计原则（强制执行 - 必须遵守）**

在执行此skill时,以下原则是**强制性的**,违反任何一项都必须重新设计方案:

- **文件数量上限 (File Count Cap - 强制约束)**
  - 单次重构生成的新文件数量**严格禁止超过 3 个**。
  - 优先尝试将策略/实现合并到现有文件中(例如作为内部类、命名空间导出或独立的函数组),而不是创建新文件。
  - **执行检查**:在生成代码前必须统计文件数量,如果超过3个,立即停止并重新设计。

## 结构化工作流程

**【强制约束】必须按顺序执行步骤1-5：五步流程与明确的成功标准使SKILL行为具有可预测性。**

### 步骤 1️⃣ : 智能代码坏味道检测

读取用户提供的代码文件(最多 5 个),进行智能分析:

**【重要】步骤1必须输出检测结果**，格式如下:
```markdown
### 步骤 1 : 智能代码坏味道检测

**[1/5] 🔄 执行中...**

[开始进行代码坏味道检测]

## 检测结果

- 检测到 [X] 处代码坏味道
- 复杂度评估: [高/中/低]
- 可维护性评估: [高/中/低]
- 扩展性评估: [高/中/低]

**[1/5] ✅ 步骤1完成**
```

**成功标准**: 输出检测结果，包括检测到的坏味道列表、复杂度评估、可维护性评估、扩展性评估

#### 代码坏味道识别方法

1. **深入理解代码逻辑**: 阅读代码,理解其业务逻辑和结构

2. **识别设计问题**: 查找以下类型的代码坏味道(但不限于这些):
    - 复杂条件分支 (大量 if-else/switch)
    - 散落的对象创建逻辑
    - 逐步构建复杂对象的逻辑
    - 手动管理多个回调或通知
    - 代码重复和高度耦合
    - 其他面向过程的设计问题

3. **评估严重程度**: 根据以下维度评估:
    - 代码复杂度 (分支数量、嵌套层级、代码行数)
    - 可维护性 (代码重复度、耦合度、可读性)
    - 扩展性 (新增功能是否需要修改现有代码)
    - 可测试性 (单元测试的难易程度)

4. **判断是否需要重构**:
    - ✅ 建议重构: 代码复杂度高、难以维护、扩展性差
    - ⚠️ 可选重构: 代码有一定复杂度,但不是必须重构
    - ❌ 不建议重构: 代码简单清晰,重构会导致过度设计

5. **预评估文件数量 (关键步骤)**:
    在提出重构方案前,先评估:
    - 当前逻辑是否只在当前文件使用?
    - 是否需要跨文件复用(3个以上其他文件)?
    - 如果单文件使用,优先考虑内部类/函数方案
    - 预估新文件数量,如果超过3个,必须重新考虑方案

#### 文件结构规划

根据推荐的设计模式,规划合理的文件结构。**必须遵守以下原则**:

1. **优先合并到现有文件**:如果逻辑只在当前文件中使用,将策略/实现作为内部类、命名空间或函数组放在原文件中
2. **严格控制新建文件数**:除非绝对必要,否则单次重构新建的文件数量**禁止超过 3 个**
3. **简化目录结构**:避免创建多层嵌套目录,优先使用扁平化结构

**推荐的文件结构示例**:

**方案1: 单文件重构(优先推荐)**
```
src/
├── [原文件].ts                # 在原文件中添加内部类/函数
```

**方案2: 最多3个新文件(仅在必要时)**
```
src/
├── [原文件].ts                # 重构后的调用方
├── [策略/实现].ts             # 策略或实现集合(如多个策略合并到一个文件)
```

**禁止的过度设计结构**:
❌ 创建 interfaces/、implementations/、factories/ 等多层目录
❌ 为每个策略/实现创建独立文件
❌ 为简单的3-5个分支创建复杂的工厂模式

#### 重构计划表格

**重要约束**:新建文件数量**不得超过3 个**

| 类型 | 路径                  | 操作说明           | 文件数量检查 |
| ---- | --------------------- | ------------------ | ------------ |
| 新建 | `path/to/file1.ts`    | 说明新建文件的用途 | 总计: X/2 个 |
| 新建 | `path/to/file2.ts`    | 说明新建文件的用途 | 总计: X/2 个 |
| 修改 | `path/to/original.ts` | 说明如何修改       | -            |
| 删除 | `path/to/old.ts`      | 说明删除原因(如有) | -            |

**文件数量验证**:
- ✅ 新建文件数量 ≤ 3: 继续执行
- ⚠️ 新建文件数量 > 3: **必须重新评估**,考虑合并到现有文件
- ❌ 新建文件数量 ≥ 5: **立即停止**,重新设计方案

**【重要】步骤4必须输出重构代码对比**，格式如下:
```markdown

**[4/5] 🔄 执行中...**

[生成重构代码并展示对比]

# 重构前后对比

[对比内容...]

**[4/5] ✅ 步骤4完成**
```

**成功标准**: 步骤4必须输出完整的重构前后代码对比，并在输出中明确显示。

**强制检查点**:在生成代码前,必须验证以下条件:

```
□ 新建文件数量 ≤ 3 个 
□ 重构后总行数增加不超过 20%
□ 没有为"未来可能的扩展"创建不必要的抽象
□ 没有创建空的接口或抽象基类(除非已有2个以上实现)
如果任何一项不满足,必须重新设计方案!
```

**重要**:

-  将重构前的原始代码和重构后的代码放在一起展示,方便用户对比理解。
- 不需要等待用户确认，直接展示重构代码对比

```markdown

**过度设计检查**:
- ✅ 新建文件数量符合约束 (≤ 3 个)
- ✅ 代码行数增长合理 (≤ 20%)
- ✅ 避免了不必要的抽象
- ✅ 优先考虑了代码复用性

**下一步操作**:
- 输入 "确认" - 生成完整的重构代码文件

```

## 重要提醒

- 始终保持代码的可读性和可维护性
- 严禁过度设计,只在必要时应用设计模式
- 确保重构后的代码通过所有测试
- 保留原始代码的备份,以便回滚
- 与团队成员沟通重构方案,确保一致性
- 优先考虑代码的实际需求和业务场景,而不是盲目应用模式
