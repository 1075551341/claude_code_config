# 技能编写指南

## 核心原则

**编写技能就是测试驱动开发应用于流程文档。**

## TDD 循环

```markdown
🔴 RED 阶段
1. 选择一个压力场景
2. 在不使用技能的情况下运行
3. 记录失败的具体表现（原文记录）

🟢 GREEN 阶段
4. 编写最小技能来解决这些具体失败
5. 重新运行场景
6. 验证失败已解决

🔵 REFACTOR 阶段
7. 检查漏洞和边界情况
8. 补充遗漏场景
9. 再次测试直到健壮
```

## 铁律

**没有先失败的测试，就没有技能。**

- 先看代理在没有技能时失败
- 然后编写技能解决那些失败
- 永远不要在测试基线行为之前编写技能

## 技能类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 技术 Technique | 具体实现方法 | 代码模板、配置示例 |
| 模式 Pattern | 心智模型和思维框架 | 调试流程、设计思路 |
| 参考 Reference | API 文档和速查 | 命令列表、参数说明 |

## 文件结构

```markdown
skills/
└── skill-name/
    ├── SKILL.md          # 必需：技能主文件
    ├── scripts/          # 可选：辅助脚本
    ├── references/       # 可选：参考文档
    └── assets/           # 可选：资源文件
```

## Frontmatter 规范

```yaml
---
name: skill-name                    # 必需：技能唯一标识
description: 用于...当...时使用。触发词：关键词1、关键词2。  # 必需：描述
---

# 注意：
# - name: 只能包含字母、数字、连字符
# - description: 必须 < 1024 字符
# - description: 以"用于"开头，描述何时使用
```

## 描述写作规则

```markdown
✅ 正确的描述：
"用于调试复杂问题，当遇到难以定位的 bug、需要系统化排查时使用。触发词：调试、debug、排查问题、定位 bug、故障排查。"

❌ 错误的描述：
"这个技能帮助调试代码，包含以下步骤：1. 观察 2. 假设 3. 验证..."
# 问题：总结了工作流程，代理会跳过阅读技能内容
```

## 关键规则

### 描述 = 何时使用，不是做什么

```markdown
❌ 不要：总结工作流程
"这个技能会帮你：1. 分析问题 2. 定位原因 3. 提出方案"

✅ 应该：描述触发条件
"用于...当需要...遇到...时使用"
```

### 语言要求

```markdown
- 技能内容优先使用中文（代码除外）
- 触发词包含中英文常见表达
- 技术术语保留英文原文
```

### 内容长度

```markdown
| 技能类型 | 建议长度 |
|----------|----------|
| 入门指南 | < 150 词 |
| 频繁加载 | < 200 词 |
| 详细参考 | 可更长 |

原则：每个字都有价值，删除废话
```

## 反模式警告

```markdown
🚨 绝对不要：

1. 在测试基线前写技能
   - 你不知道代理实际会怎么失败
   - 技能可能解决不存在的问题

2. 在描述中总结工作流程
   - 代理会跳过阅读内容
   - 描述只说明何时使用

3. 多语言示例
   - 一个优秀示例足够
   - 多语言示例增加噪音

4. 在技能中写代码流程图
   - 代码应直接可运行
   - 流程图是冗余
```

## 技能模板

```markdown
---
name: skill-name
description: 用于[目的]，当[触发条件]时使用。触发词：[关键词列表]。
---

# [技能名称]

## 核心原则

[一句话说明核心思想]

## 使用场景

[什么情况下使用这个技能]

## [主要内容]

[具体内容，可以是步骤、模板、示例等]

## 最佳实践

[关键注意事项]

## 检查清单

- [ ] 检查项1
- [ ] 检查项2
```

## 测试技能

```markdown
## 测试流程

1. 创建测试场景
   - 选择一个典型的使用场景
   - 准备必要的输入和预期输出

2. 不使用技能运行
   - 记录代理的行为
   - 记录失败或不理想的地方

3. 使用技能运行
   - 加载技能
   - 观察代理行为变化
   - 验证问题是否解决

4. 边界测试
   - 测试极端情况
   - 测试技能的适用边界
```

## 评估标准

```markdown
## 好技能的标准

1. 触发准确
   - 该触发时不漏触发
   - 不该触发时不误触发

2. 内容有效
   - 解决了实际问题
   - 提供了可执行指导

3. 长度适中
   - 没有冗余内容
   - 关键信息突出

4. 维护简单
   - 结构清晰
   - 易于更新
```
