# 实施计划编写

## 核心原则

**设计批准后，创建可执行的分步实施计划，每步独立验证。**

---

## 适用场景

- 设计文档已批准
- 功能开发启动前
- 复杂任务分解
- 多文件修改规划

---

## 计划结构

### 必需内容

```markdown
# 实施计划：[功能名称]

## 上下文
- 设计文档：[链接]
- 目标：[一句话]

## 实施步骤

### 步骤 1：[名称]
- 文件：[具体路径]
- 操作：[创建/修改/删除]
- 内容：[具体做什么]
- 验证：[如何确认完成]

### 步骤 2：[名称]
...

## 风险点
- [潜在问题]
- [应对方案]

## 完成标准
- [验收条件]
```

---

## 步骤设计原则

### 独立可验证

```
每个步骤：
- 可独立执行
- 完成后可验证
- 不依赖后续步骤
- 失败时可定位
```

### 最小修改集

```
原则：
- 一个步骤改变最少数文件
- 相关改变放一起
- 不捆绑无关修改
```

### 检查点设置

```
关键节点：
- 基础设施搭建完成
- 核心功能实现完成
- 集成测试通过
- 文档更新完成

每步后运行验证命令。
```

---

## 执行流程

### 执行前

```
1. 确认设计文档已批准
2. 检查依赖是否就绪
3. 创建开发分支
```

### 执行中

```
1. 按顺序执行每步
2. 每步完成后验证
3. 验证失败则停止调试
4. 不跳过验证继续
```

### 执行后

```
1. 运行全部测试
2. 更新文档
3. 创建提交
4. 准备合并
```

---

## 分支策略

```
主分支 → 功能分支 → 开发 → 测试 → 合回主分支

命名：feature/<功能名>
提交：每步完成后提交
合并：测试全部通过后
```

---

## 验证命令模板

| 类型 | 命令 |
|------|------|
| 类型检查 | `tsc --noEmit` |
| 单元测试 | `npm test` 或 `pytest` |
| 集成测试 | `npm run test:integration` |
| 代码检查 | `eslint` 或 `ruff check` |
| 构建验证 | `npm run build` |

---

## 常见问题处理

| 问题 | 处理 |
|------|------|
| 步骤失败 | 停止，调用 `systematic-debugging` |
| 设计需调整 | 回到 `design-brainstorming` |
| 发现新需求 | 记录，当前计划完成后处理 |
| 依赖阻塞 | 标记，等待或寻求帮助 |
