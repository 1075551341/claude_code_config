## 编写原则

### 1. SMART 目标

```
S - Specific   具体明确
M - Measurable 可衡量
A - Achievable 可实现
R - Relevant   相关性
T - Time-bound 有时限
```

### 2. 步骤粒度

```
✅ 合适粒度：一个步骤可在 1-3 天内完成
❌ 过粗：一个步骤需要 2 周
❌ 过细：一个步骤只需 1 小时
```

### 3. 依赖关系

```
使用 DAG（有向无环图）表示依赖：
- 明确哪些任务可以并行
- 标识关键路径
- 避免循环依赖
```

## 计划类型

### 功能开发计划

```markdown
## 功能：用户登录

### 技术方案

- 前端：React + TypeScript
- 后端：Node.js + Express
- 认证：JWT

### 实施步骤

1. 数据库设计（1天）
2. 后端 API 开发（2天）
3. 前端页面开发（2天）
4. 联调测试（1天）
5. 上线部署（0.5天）

### 验收标准

- [ ] 用户可通过邮箱密码登录
- [ ] 登录失败显示明确错误提示
- [ ] Token 有效期 7 天，支持刷新
```

### 重构计划

```markdown
## 重构：订单模块拆分

### 当前问题

- 订单服务代码 5000+ 行
- 耦合度高，修改风险大

### 目标架构

[架构图]

### 迁移策略

采用绞杀者模式，逐步替换：

1. 新功能在新服务实现
2. 旧功能逐步迁移
3. 最终下线旧服务

### 回滚方案

每个阶段都可独立回滚
```

### 上线计划

```markdown
## 上线：V2.0 版本

### 上线前检查

- [ ] 所有测试通过
- [ ] 性能测试达标
- [ ] 文档更新完成
- [ ] 回滚脚本准备

### 上线步骤

1. 备份数据库（10:00）
2. 停止旧服务（10:05）
3. 部署新版本（10:10）
4. 数据迁移（10:15）
5. 启动新服务（10:30）
6. 验证测试（10:45）
7. 开放流量（11:00）

### 回滚触发条件

- 错误率 > 1%
- 响应时间 > 2s
- 关键功能不可用

### 回滚步骤

1. 切换流量到旧版本
2. 恢复数据库
3. 重启旧服务
```

## 计划评审检查清单

- [ ] 目标清晰、可衡量
- [ ] 步骤完整、无遗漏
- [ ] 依赖关系明确
- [ ] 时间估算合理
- [ ] 风险已识别
- [ ] 验收标准明确
- [ ] 有回滚/应急预案
