# skill-reviewer的审查规则集

## 核心检查项

### 1. 命名规范检查

#### 1.1 烤串命名法检查 (G.NAM.01)
**要求**：使用烤串命名法（kebab-case），即使用小写英文字母、数字和连字符，连字符不能在首部和尾部，最多64个字符。

**检查内容**：
- 检查skill名称是否只包含小写字母、数字和连字符
- 检查连字符是否不在首部和尾部
- 检查总长度是否不超过64个字符

**反例**：
```
name: "React_Component_Tester"
name: "skill_reviewer"
name: "ValidSkillName"
```

**正例**：
```
name: "react-component-tester"
name: "skill-reviewer"
name: "valid-skill-name"
```

**修改建议**：
- 将大写字母转换为小写字母
- 将下划线(_)替换为连字符(-)
- 将驼峰式命名转换为烤串式命名
- 确保字母和数字交替排列时使用连字符分隔

#### 1.2 功能性命名模式检查 (G.NAM.02)
**要求**：使用功能性命名模式，采用[Action]-[Object] 或 [Context]-[Task] 结构。

**检查内容**：
- 分析名称是否包含动作对象或上下文任务结构
- 评估名称的功能清晰度和表达准确性

**反例**：
```
name: "generative-or-boilerplate-tasks"
name: "advice-heavy-skills"
name: "complex-system-tool"
```

**正例**：
```
name: "creating-unit-tests"
name: "skills-creator"
name: "code-reviewer"
name: "database-migration"
```

**修改建议**：
- 确定skill的核心功能，选择明确描述功能的动词
- 选择能够准确描述处理对象或场景的名词
- 确保名称简洁明了，避免冗余词汇

#### 1.3 避免语义噪音检查 (G.NAM.03)
**要求**：避免使用Agent/SubAgent/MCP/assistant/Tool等产生"语义噪音"的词汇。

**检查内容**：
- 检查名称中是否包含语义噪音词汇
- 评估名称的准确性和专业性

**反例**：
```
name: "email-agent-skill"
name: "file-processor-tool"
name: "api-assistant"
```

**正例**：
```
name: "drafting-emails"
name: "file-processor"
name: "api-generator"
```

**修改建议**：
- 移除不需要的功能性修饰词
- 专注于描述实际操作或结果
- 使用简洁、专业的词汇

#### 1.4 子分类后缀检查 (G.NAM.04)
**要求**：如果在同一个领域拥有多项技能，使用后缀来表明子分类。

**检查内容**：
- 检查同领域skill是否使用统一的后缀模式
- 评估后缀的描述性和一致性

**反例**：
```
name: "db-postgres"
name: "db-mongo"
name: "db-sql"
```

**正例**：
```
name: "db-postgres-optimize"
name: "db-postgres-migration"
name: "db-mongodb-query"
name: "db-mongodb-design"
```

**修改建议**：
- 识别skill的具体功能或用途
- 添加能够描述主要功能的动词性后缀
- 确保同一领域的skill保持命名一致性

### 2. 目录结构检查

#### 2.1 目录与skill名称一致性检查 (G.FMT.01)
**要求**：skill技能包名称必须与skill的名称保持一致。

**检查内容**：
- 检查目录名称是否与SKILL.md中的name字段一致

**反例**：
```
目录名: "codeReviewer"
SKILL.md name: "skill-reviewer"
```

**正例**：
```
目录名: "code-reviewer"
SKILL.md name: "code-reviewer"
```

**修改建议**：
- 确保目录名称使用烤串命名法
- 确保目录名称与skill名称完全一致

#### 2.2 脚本目录检查 (G.FMT.02)
**要求**：确定性高和重复性强的任务以脚本的形式存放在scripts目录中，如果没有脚本，可以没有scripts目录。

**检查内容**：
- 检查是否有脚本，如何有脚本，必须存在scripts目录，而不是与skill.md在同一级目录
- 验证scripts目录中是否包含适当类型的脚本
- 评估脚本的命名规范性

**反例**：
```
/
├── setup.sh
├── skill.md
├── validate.py
└── cleanup.sh
```

**正例**：
```
/scripts/
├── setup.sh
├── validate.py
└── cleanup.sh
```

**修改建议**：
- 创建scripts目录用于存放确定性高的脚本
- 将重复性任务脚本化并放入scripts目录
- 为脚本使用规范的命名

### 3. 文件格式检查

#### 3.1 命名语言要求检查 (G.FMT.05)
**要求**：skill name和目录名称必须使用英文。

**检查内容**：
- 检查目录下一定有SKILL.md文件，该文件名称不能带前缀或后缀
- 检查SKILL.md中的name字段是否为英文
- 验证目录名称是否符合英文命名规范

**反例**：
```
name: "代码审查工具"
目录名: "代码审查"
```

**正例**：
```
name: "code-reviewer"
目录名: "code-reviewer"
其他文件: "代码检查规范.md" 或 "code-standards.md"
```

**修改建议**：
- 将SKILL.md中的name字段和目录名转换为规范的英文烤串命名
- 确保文件名清晰表达文件功能

### 4. 内容要求检查

#### 4.1 有效描述检查 (G.EXP.01)
**要求**：Skill描述最多1024个字符，且不能为空。描述该技能能够完成的功能以及使用场景。

**检查内容**：
- 检查description字段是否为空
- 验证描述字符数是否超过1024
- 评估描述的功能性和场景清晰度

**反例**：
```
description: ""
description: "这是一个工具" (过于简单)
```

**正例**：
```
description: "执行全面的代码质量验证，包括代码合规性检查、架构合理评估、性能优化建议、安全漏洞分析、可维护性评估等，输出结构化的审查报告并提供改进建议。"
```

**修改建议**：
- 添加详细的功能描述
- 包含具体的使用场景
- 确保描述不超过1024个字符
- 使用专业、准确的技术语言

#### 4.2 术语一致性检查 (G.EXP.02)
**要求**：在skill的文件内容描述中使用一致的术语。

**检查内容**：
- 检查术语使用的一致性
- 识别术语混用或歧义

**反例**：
```
在文件中同时使用"数据结构"和"数据结构"的混写
在文件中同时使用"验证"和"校验"表达相同概念
```

**正例**：
```
统一使用"数据结构"而非"数据结构"
统一使用"验证"概念，避免混用"校验"
```

**修改建议**：
- 创建术语表确保用词一致性
- 在整个文档中使用统一的术语
- 避免使用多义词或易混淆的词汇

#### 4.3 输入输出前置条件明确性检查 (G.EXP.03)
**要求**：明确skill的输入、输出、前置条件。

**检查内容**：
- 如果有输入参数，检查是否明确说明输入参数要求
- 验证输出格式的定义是否清晰
- 评估前置条件的完整性

**反例**：
```
缺少输入参数说明
未说明前置条件
输出格式描述模糊
```

**正例**：
```
输入：要检查的skill路径（必需）
输出：结构化的合规性检查报告
前置条件：目标skill路径存在且可访问
```

**修改建议**：
- 添加清晰的输入参数说明
- 详细描述输出格式和内容
- 明确列出所有必要的前置条件
- 为复杂参数提供示例值

#### 4.4 外部依赖声明检查 (G.EXP.04)
**要求**：如果对外部有依赖，需要正确声明和引用外部依赖。

**检查内容**：
- 检查是否声明了必要的外部依赖
- 验证依赖的版本要求
- 评估依赖声明的准确性

**反例**：
```
缺少对必要系统的依赖声明
依赖版本要求不明确
```

**正例**：
```
dependencies:
  - "codemate: >=1.0.0"
  - "python: >=3.8"
```

**修改建议**：
- 添加所有必要的外部依赖声明
- 明确指定依赖的最低版本要求
- 避免包含不必要或过时的依赖

#### 4.5 异常处理机制检查 (G.EXP.05)
**要求**：提供健壮的异常处理机制。

**检查内容**：
- 检查是否描述了异常场景
- 验证异常处理策略的完整性
- 评估恢复机制的可行性

**反例**：
```
缺少对常见异常的说明
未提供错误处理建议
```

**正例**：
```
异常处理：
- 文件不存在：请检查路径是否正确
- 权限不足：请确保具有读取权限
- 系统错误：请检查运行环境配置
```

**修改建议**：
- 添加常见异常场景的处理指南
- 提供明确的错误信息说明
- 建议适当的恢复措施

#### 4.6 资源清理检查 (G.EXP.06)
**要求**：执行完成后正确清理临时资源。

**检查内容**：
- 检查是否提及资源清理
- 验证清理机制的完整性
- 评估资源管理的全面性

**反例**：
```
未提及临时文件清理
资源泄漏风险说明不足
```

**正例**：
```
资源管理：
- 自动清理临时创建的文件
- 确保网络连接正确关闭
- 释放系统资源占用
```

**修改建议**：
- 添加明确的资源清理说明
- 说明自动和手动清理的方式
- 强调资源管理的重要性

### 5. 最佳实践检查

#### 5.1 文档完整性与结构化检查（G.PRA.01）
**要求**：确保文档完整且结构清晰。

**检查内容**：
- 验证基础文件的存在，必须包含CHANGELOG.md、LICENSE.md、README.md这三个文件
- 检查文档目录结构的合理性

**反例**：
```
skill的目录中缺少LICENSE.md文件
```

**正例**：
```
目录结构：
/skill-name/
├── SKILL.md         # SKILL定义和规范
├── CHANGELOG.md     # 版本变更记录
├── LICENSE.md       # 许可证信息
├── README.md        # 用户指南
└── scripts/         # 脚本文件
```

**修改建议**：
- 确保所有必要的文档文件存在，CHANGELOG.md、LICENSE.md、README.md都是给用户看的，让用户更了解本skill。
- 组织合理的目录结构

#### 5.2 错误处理和日志检查（G.PRA.02）
**要求**：提供完善的错误处理和日志记录机制。

**检查内容**：
- 检查错误处理策略的完整性
- 评估日志记录的充分性
- 验证调试信息的可用性

**反例**：
```
错误信息模糊不清
缺少必要的日志记录
```

**正例**：
```
错误处理：
- 详细的错误代码和描述
- 用户友好的错误信息
- 详细的调试日志记录
```

**修改建议**：
- 实现分级的错误处理机制
- 添加详细的日志记录
- 提供清晰的错误提示信息
