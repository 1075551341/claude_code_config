# 代码审查工作流

## 核心能力

**请求审查、响应反馈、完成PR合并全流程指导。**

---

## 适用场景

- 请求代码审查
- 响应审查意见
- PR 合并流程
- 审查最佳实践

---

## 请求代码审查

### 审查前自查

```markdown
## 提交前检查清单

□ 代码编译通过
□ 测试全部通过
□ 无明显代码问题
□ 文档已更新
□ commit message 规范
□ 变更范围明确
```

### PR 描述模板

```markdown
## 变更概述
[一句话描述本次变更目的]

## 变更内容
- [具体变更1]
- [具体变更2]

## 测试情况
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 手动测试验证

## 截图/演示
[如有UI变更，附上截图]

## 相关链接
- Issue: #xxx
- 设计文档: [链接]

## 审查重点
请特别关注：[指定需要重点审查的部分]
```

### 审查者选择

```
选择原则：
1. 熟悉该代码领域
2. 有时间进行审查
3. 非变更直接作者
4. 通常2-3人足够
```

## 响应审查反馈

### 反馈分类处理

| 类型 | 处理方式 |
|------|----------|
| 必须修改 | 立即修复，标注已解决 |
| 建议优化 | 评估后决定，说明理由 |
| 问题讨论 | 在评论中解释或讨论 |
| 已过时 | 标注不需要处理 |

### 回复模板

```markdown
## 对于必须修改
✅ 已修复，commit: abc123

## 对于建议
采纳，已修改 / 暂不采纳，原因：xxx

## 对于讨论
解释：xxx / 同意，已按建议修改
```

### 避免行为

```
不要：
- 忽略审查意见
- 防御性回复
- 无解释拒绝
- 拖延响应时间

应该：
- 及时响应（24小时内）
- 清晰说明决策
- 感谢审查者
- 保持专业态度
```

## 审查完成流程

### 合并前确认

```markdown
□ 所有评论已处理
□ 审查者已批准
□ CI/CD 通过
□ 无合并冲突
□ 分支已同步
```

### 合并方式选择

| 方式 | 适用场景 |
|------|----------|
| Merge commit | 保留完整历史 |
| Squash merge | 压缩为单次提交 |
| Rebase merge | 线性历史 |

### 合并后操作

```bash
# 删除功能分支
git branch -d feature/xxx
git push origin --delete feature/xxx

# 同步本地主分支
git checkout main
git pull origin main
```

## 审查者指南

### 审查重点

```markdown
## 正确性
- 逻辑是否正确
- 边界情况处理
- 错误处理完整

## 安全性
- 无安全漏洞
- 敏感数据处理
- 权限验证

## 性能
- 无性能问题
- 算法复杂度合理
- 资源使用优化

## 可维护性
- 代码清晰可读
- 命名规范
- 注释适当

## 测试
- 测试覆盖充分
- 测试用例有效
```

### 审查评论规范

```markdown
# 好 comment 示例
"这里可能存在空指针问题，建议添加 null 检查"
"性能优化：可以使用 Map 代替 List 查找，O(n) → O(1)"

# 避免
"代码有问题" （不具体）
"改成 xxx" （无解释）
```
