# Skill名称: frontend-refactor-proposer

[TOC]

## 概述

**前端重构设计 (frontend-refactor-proposer)** 是专为前端开发者设计的局部重构决策助手。当用户提出具体文件（如 .ts、.js、.html、.vue、.jsx）存在“圈复杂度高”、“难以维护”、“无法测试”、“逻辑耦合”、“代码复杂”、“逻辑不清晰”、“无法写单元测试”、“全局依赖”时，本技能不会直接生成代码，而是强制输出【保守/均衡/激进】三套差异化重构策略，并提供多维度的量化对比分析与最终决策建议。

本技能仅支持单文件或微模块级别的深度分析，不支持全项目架构改造。它通过智能诊断引擎将用户自然语言描述转化为专业技术缺陷，并根据用户的具体约束条件（工期限制、团队背景、优先目标等）动态调整评价权重，最终推荐最适合当前场景的重构方案。

## 功能特性

本技能提供以下核心功能：

- **智能诊断引擎**：将用户的自然语言描述转化为专业技术缺陷，例如将"改一处坏多处"识别为高耦合、违反开放封闭原则 (OCP)，将"看不懂/像面条"识别为高认知复杂度、控制流混乱，将"没法测"识别为硬编码依赖、隐式副作用

- **三套差异化重构方案**：
  - **方案 A：保守优化型 **：重命名、提取子函数、守卫子句、配置化硬编码，不改变文件结构
  - **方案 B：均衡重构型 **：引入设计模式（策略/工厂）、拆分内部类/组件、简易依赖注入，可能拆分为 2-3 个小文件
  - **方案 C：激进重塑型 **：使用新范式（如 State Machine）重写核心逻辑、彻底解耦内部依赖，完全重写该模块

- **动态权重分配**：根据用户约束场景自动调整评价维度权重，包括工期极紧/线上救火、无法写测试/逻辑混乱、核心模块/长期维护、团队能力弱/技术栈旧等场景

- **多维度量化评价**：对每个方案进行价值、难度、风险、收益的星级评价，以及可维护性、可测试性、可靠性、可扩展性、安全性、性能的白盒评价（1-10分）

- **决策支持系统**：基于加权计算自动推荐最优方案，并提供详细的决策依据和理由说明

## 前置条件与配置

本技能无外部依赖，无需任何特殊安装配置。只需确保 Agent 环境能够访问技能文件即可使用。

## 本地测试与使用

本技能不涉及脚本执行，无需本地测试脚本。使用方式如下：

1. 在 Agent 中选中需要分析的前端文件或模块
2. 使用触发提示词唤醒技能
3. 技能将自动分析问题并提供三种重构方案及决策建议

## 触发提示词与用户场景

### 使用方式

必须结合文件/模块使用，在 Agent 中选中有问题的组件。

### 支持的场景 (In-Scope)

本技能专门针对文件级 (File-Level) 或 模块级 (Module-Level) 的局部问题：

- **单文件复杂度过高**：某个 .vue、.tsx、.js 文件行数过多、圈复杂度高、逻辑混乱
- **特定功能难以维护**：某个具体函数或类存在"散弹式修改"（改一处坏多处）、逻辑耦合严重
- **单元测试受阻**：因硬编码依赖、全局状态引用导致无法对特定文件编写测试
- **代码可读性差**：命名晦涩、控制流嵌套过深、缺乏语义化拆分，导致新人难以理解
- **局部技术债**：某个文件中混用了旧范式（如 Class 组件与新 Hooks 混用），需要统一风格
- **重复代码消除**：同一文件或紧密关联的几个文件之间存在明显的复制粘贴逻辑

### 触发示例

#### 场景 1：修复 codecheck 指标

**提示词**：
```
这个 vue 文件圈复杂度高达 50，帮我进行重构方案设计和方案分析。
```

**期望行为**：Agent 成功唤醒 SKILL，针对用户问题提供三种重构方案，给出重构方案的评价，可以结合用户实际情况（工期、团队技术背景、优先目标）等给出最终推荐方案。

#### 场景 2：描述具体代码痛点

**提示词**：
```
这个 vue 文件难以维护，帮我进行重构方案设计和方案分析。
```

**期望行为**：Agent 成功唤醒 SKILL，针对用户问题提供三种重构方案，给出重构方案的评价，可以结合用户实际情况（工期、团队技术背景、优先目标）等给出最终推荐方案。

#### 场景 3：测试受阻问题

**提示词**：
```
这个组件没法写单元测试，帮我进行重构方案设计和方案分析。
```

**期望行为**：Agent 成功唤醒 SKILL，重点解决可测试性问题，提供依赖注入等重构方案。

### 不支持的场景 (Out-of-Scope)

若用户请求涉及以下宏观问题，技能将拒绝并提供引导建议：

- **全项目技术栈迁移**：如 "如何将整个项目从 Vue2 升级到 Vue3？" 或 "从 Webpack 迁移到 Vite"
- **全局架构重构**：如 "如何重新设计整个应用的状态管理架构？" 或 "如何拆分微前端？"
- **跨模块依赖治理**：涉及数十个文件循环依赖的清理（除非用户能提供具体的核心冲突文件对）
- **构建/部署流程优化**：如 CI/CD 流水线改造、打包体积全局优化策略
- **设计规范统一**：如 "如何统一整个项目的代码风格？"（应推荐使用 Linter/Prettier 工具而非人工重构方案）

## 输入与输出规范

### 输入规范

#### 必填项

1. **代码片段/文件内容**
   - **严格数量限制**：最多支持 **5** 个文件（总代码量建议控制在 1000 行以内，以确保分析深度）
   - **格式**：支持 TS、JS、Vue、React、JSX 等

2. **问题描述 (Problem Statement)**
   用户可以用自然语言描述当前痛点，支持混合模式：
   - **量化指标类**："圈复杂度高达 50"、"重复代码率 40%"
   - **定性体验类**：
     - 改一个地方崩三个地方 (Shotgun Surgery)
     - 根本没法写单元测试，依赖全是全局单例
     - 逻辑太散，新人看不懂
     - 虽然指标正常，但扩展新业务极其困难

#### 选填项

1. **time_constraint**：工期限制（如 "2 天内"、"无限制"）
2. **team_context**：团队背景（如 "团队不熟悉函数式编程"、"缺乏自动化测试"）
3. **goal_priority**：优先目标（如 "首要解决可测试性"、"首要降低认知负荷"）

### 输出规范

技能将严格按照以下结构输出，输出结果在Agent Chat聊天窗口中显示：

#### 1. 诊断摘要
- **核心痛点**：将用户描述转化为专业术语
- **影响范围**：明确指出仅影响哪些文件/模块，强调局部性

#### 2. 三种重构方案

**方案 A：保守优化型**

- 核心策略：详细阐述重构策略，指出具体的代码行数以及改动方法
- 针对性解决：痛点改善程度
- 预期指标变化：数据预估
- 多维评价：价值、难度、风险、收益的星级评价
- 白盒评价：可维护性、可测试性、可靠性、可扩展性、安全性、性能的评分（1-10分）
- 专家点评：一句话建议

**方案 B：均衡重构型 **

- 核心策略：详细阐述重构策略，指出具体的代码行数以及改动方法
- 针对性解决：痛点改善程度
- 预期指标变化：数据预估
- 多维评价：价值、难度、风险、收益的星级评价
- 白盒评价：可维护性、可测试性、可靠性、可扩展性、安全性、性能的评分（1-10分）
- 专家点评：一句话建议

**方案 C：激进重塑型 **

- 核心策略：详细阐述重构策略，指出具体的代码行数以及改动方法
- 针对性解决：痛点改善程度
- 预期指标变化：数据预估
- 多维评价：价值、难度、风险、收益的星级评价
- 白盒评价：可维护性、可测试性、可靠性、可扩展性、安全性、性能的评分（1-10分）
- 专家点评：一句话建议

#### 3. 综合决策建议
- **当前场景判定**：明确指出用户属于哪种约束场景
- **权重调整说明**：说明为何某些维度权重被放大或降低
- **最终推荐**：经过加权计算后得分最高的方案（禁止默认选 B）
- **理由**：结合约束条件，解释为何此局部方案最优

## 限制与已知问题

### 关键限制

- **文件数量熔断机制**：单次请求严格限制为 5 个文件。若上传超过 5 个文件，技能将立即终止分析并提示用户优先选择复杂度最高的 5 个文件，或分批次提交。

- **代码量限制**：总代码量建议控制在 1000 行以内，以确保分析深度和准确性。

- **范围限制**：仅支持单文件或微模块级别的分析，不支持全项目架构改造。若问题明显超出单文件/小模块范畴，将触发"不支持场景"响应策略。

- **无代码生成**：本技能只做重构方案设计和方案分析，不做代码改动。

### 边界情况

- **问题描述过于模糊**：若用户提供的问题描述不够具体，技能可能无法准确识别核心技术缺陷，建议用户提供更详细的上下文信息。

- **选填项缺失**：若未提供工期限制、团队背景、优先目标等选填项，技能将按默认场景（无明确约束）进行评价，权重分配为综合性价比。

- **跨文件依赖复杂**：虽然支持最多 5 个文件，但如果这些文件之间存在复杂的循环依赖或跨模块引用，分析准确性可能受到影响。

## 非功能性指标

### 性能指标

- **平均延迟**：~10-15 秒每次执行，高度依赖于代码量和复杂度
- **Token 消耗**：较高。每次标准执行预计消耗 ~2000-3000 prompt tokens 和 ~1500-2500 completion tokens
- **可靠性**：90%+ 的成功率。主要失败场景包括文件数量超限、问题描述过于模糊、超出服务范围等情况

### 质量指标

- **诊断准确性**：基于自然语言描述的技术缺陷识别准确率约为 85%
- **方案适用性**：生成的三种方案能够覆盖 90% 以上的实际重构场景
- **决策合理性**：基于动态权重分配的推荐方案符合用户约束条件的准确率约为 80%
