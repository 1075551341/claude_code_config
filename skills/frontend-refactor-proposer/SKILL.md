# 前端重构设计

针对前端项目，本技能帮助你基于用户问题描述与代码内容，生成多个可行的重构方案，并提供决策辅助。它会深入分析前端问题，从多个维度评估方案，帮助你做出最佳选择。

## 适用场景
✅ 支持场景 (In-Scope)  
   本 Skill 专门针对文件级 (File-Level) 或 模块级 (Module-Level) 的局部问题：  
      1、 单文件复杂度过高: 某个 .vue, .tsx, .js 文件行数过多、圈复杂度高、逻辑混乱。  
      2、 特定功能难以维护: 某个具体函数或类存在“散弹式修改”（改一处坏多处）、逻辑耦合严重。  
      3、 单元测试受阻: 因硬编码依赖、全局状态引用导致无法对特定文件编写测试。  
      4、 代码可读性差: 命名晦涩、控制流嵌套过深、缺乏语义化拆分，导致新人难以理解。  
      5、 局部技术债: 某个文件中混用了旧范式（如 Class 组件与新 Hooks 混用），需要统一风格。  
      6、 重复代码消除: 同一文件或紧密关联的几个文件之间存在明显的复制粘贴逻辑。  

❌ 不支持场景 (Out-of-Scope)  
   若用户请求涉及以下宏观问题，Skill 应拒绝并提供引导建议（如建议进行架构评审）：  
      1、 全项目技术栈迁移: 如 “如何将整个项目从 Vue2 升级到 Vue3？”或 “从 Webpack 迁移到 Vite”。  
      2、 全局架构重构: 如 “如何重新设计整个应用的状态管理架构？”或 “如何拆分微前端？”。  
      3、 跨模块依赖治理: 涉及数十个文件循环依赖的清理（除非用户能提供具体的核心冲突文件对）。  
      4、 构建/部署流程优化: 如 CI/CD 流水线改造、打包体积全局优化策略。  
      5、 设计规范统一: 如 “如何统一整个项目的代码风格？”（应推荐使用 Linter/Prettier 工具而非人工重构方案）。

## 工作流程

## 🚨 核心约束与前置校验 

### 第一步：问题收集与分析
****
首先需要明确用户想要解决的问题和目标。

**必填项**  
   - 代码片段/文件内容:  
      - **严格数量限制**: 最多支持**5**个文件（总代码量建议控制在 1000 行以内，以确保分析深度）。  
      - 格式: 支持 TS, JS, Vue, React, JSX 等。  
   - 问题描述 (Problem Statement):  
     用户可以用自然语言描述当前痛点，支持混合模式：  
     - 量化指标类: "圈复杂度高达 50", "重复代码率 40%"。  
     - 定性体验类:  
       - 改一个地方崩三个地方 (Shotgun Surgery) 
       - 根本没法写单元测试，依赖全是全局单例 
       - 逻辑太散，新人看不懂  
       - 虽然指标正常，但扩展新业务极其困难

**选填项**  
   - time_constraint: 工期限制 (如 "2 天内", "无限制")  
   - team_context: 团队背景 (如 "团队不熟悉函数式编程", "缺乏自动化测试")  
   - goal_priority: 优先目标 (如 "首要解决可测试性", "首要降低认知负荷")  

### 第二步：范围校验
  **文件数量熔断机制**
    - **最大限制**: 单次请求**严格限制为 5 个文件**。
    - **执行逻辑**:
      - **Step 0 [强制检查]**: 在接收用户请求的第一时间，统计用户上传的文件数量 `N`。
      - **IF N > 5**:
        - **立即终止**后续所有分析、重构方案生成步骤。
        - **输出专用错误提示**: "⚠️ **文件数量超限**: 本技能单次最多支持分析 **5 个文件**，检测到您上传了 {{N}} 个文件。\n\n💡 **建议操作**: \n1. 请优先选择复杂度最高、问题最核心的 5 个文件重新提交。\n2. 或将文件分为多批次（如 Batch 1, Batch 2）依次发送进行分析。"
        - **禁止**尝试只处理前 5 个而忽略其余文件（避免用户困惑）。
      - **IF N ≤ 5**: 继续执行后续的诊断与方案生成流程。
  **问题宏观程度**
    - 若问题明显超出单文件/小模块范畴（如涉及全局架构变更），触发 “不支持场景” 响应策略，引导用户聚焦具体文件。
    - 通过: 若确认为局部问题，进入诊断流程。

### 第三步：智能诊断引擎
将“用户自然语言描述”转化为“专业技术缺陷”，并自主验证：  
- 用户说：“改一处坏多处” → AI 识别: 高耦合, 违反开放封闭原则 (OCP)。
- 用户说：“看不懂/像面条” → AI 识别: 高认知复杂度, 控制流混乱。
- 用户说：“没法测” → AI 识别: 硬编码依赖, 隐式副作用。

### 第四步：多方案生成策略
始终生成 3 套 差异化方案，仅限局部代码改动：  
| 方案类型 | 策略侧重 | 典型手段 (局部) | 适用场景 |
| :--- | :--- | :--- | :--- |
| **方案 A: 保守优化型**(Conservative) | **微创手术** | 重命名、提取子函数、守卫子句、配置化硬编码。**不改变文件结构**。 | 工期极紧、测试覆盖低、求稳。 |
| **方案 B: 均衡重构型**(Balanced) | **结构重组** | 引入设计模式（策略/工厂）、拆分内部类/组件、简易依赖注入。**可能拆分为 2-3 个小文件**。 | 常规迭代、有一定测试基础、追求可维护性。 |
| **方案 C: 激进重塑型**(Radical) | **局部重写** | 使用新范式（如 State Machine）重写核心逻辑、彻底解耦内部依赖。**完全重写该模块**。 | 核心模块专项重构、测试覆盖高、团队能力强。 |

### 第五步：多维评价模型
**动态权重分配**

| 用户约束场景                                                 | 高权重维度 (Weight × 2.0)  | 低权重维度 (Weight × 0.5) | 倾向性指引                                   |
| :----------------------------------------------------------- | :------------------------- | :------------------------ | :------------------------------------------- |
| **工期极紧 / 线上救火** (e.g., "明天上线", "紧急修复")       | **实施成本**, **风险**     | 长期收益, 代码完美度      | ⬅️ 强力指向 **方案 A (保守)**                 |
| **无法写测试 / 逻辑混乱** (e.g., "没法测", "改一处坏多处")   | **可测试性**, **解耦度**   | 实施速度, 代码行数        | ⬅️ 强力指向 **方案 B (均衡)** 或 **C (激进)** |
| **核心模块 / 长期维护** (e.g., "核心计费逻辑", "新人难上手") | **可维护性**, **认知负荷** | 短期实施成本              | ⬅️ 强力指向 **方案 C (激进)**                 |
| **团队能力弱 / 技术栈旧** (e.g., "团队不懂函数式", "老项目") | **技术门槛**, **兼容性**   | 架构先进性                | ⬅️ 指向 **方案 A** 或 **简化版 B**            |
| **无明确约束 (默认)**                                        | **综合性价比** (均衡)      | -                         | ➡️ 指向 **方案 B**                            |

## 示例交互 

### **✅ 正确示例 (In-Scope)**

**用户**: "请分析 `PaymentModal.vue`。问题是每次加新渠道都要改五个地方，而且没法测，因为依赖了全局 Store。工期 3 天。"
**Skill**: (输出针对该文件的三种重构方案，重点解决 OCP 和依赖注入问题)

### **❌ 错误示例处理 (Out-of-Scope)**

**用户**: "我们要把整个项目从 jQuery 迁移到 React，顺便把所有老代码重构一遍，给个方案。"
**Skill**:

> ⚠️ **超出服务范围提示**
>
> 本专家专注于**单文件或少数相关模块的局部重构方案**（如解决某个组件的复杂度高、难测试问题）。
>
> 您提出的 **'全项目技术栈迁移 (jQuery -> React)'** 属于**全局架构重构**范畴，涉及构建流程、路由体系、状态管理等宏观决策，超出了本 Skill 的分析边界。
>
> **建议**:
>
> 1. 组织专项架构评审会议。
> 2. 制定分阶段迁移路线图。
> 3. **如果您有迁移过程中遇到的具体‘痛点文件’**（例如某个逻辑极其复杂的 jQuery 插件难以转换为 React 组件），请提供该文件代码，我可以为您提供**针对该文件的转换与重构方案**。"
