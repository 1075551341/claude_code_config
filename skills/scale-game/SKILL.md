# 尺度游戏

## 核心原则

```
极端情况 → 揭示本质 → 发现边界 → 验证假设
```

## 方法论

### 步骤1: 定义尺度维度

识别可以扩展的关键维度：

```
尺度维度:
- 用户规模: [当前 → 极小/极大]
- 数据量: [当前 → 极小/极大]
- 时间: [当前 → 极短/极长]
- 复杂度: [当前 → 极简/极繁]
- 资源: [当前 → 极少/极多]
```

### 步骤2: 极端推演

将每个维度推到极端：

```
维度: [名称]
极小值: [极端情况A]
极大值: [极端情况B]
中间值: [当前情况]
```

### 步骤3: 分析影响

观察极端情况下的系统行为：

```
极端情况 → 系统反应 → 根本约束
```

**关键问题**：
- 在极端情况下什么会崩溃？
- 什么假设不再成立？
- 什么是真正重要的？
- 边界在哪里？

### 步骤4: 提取洞察

从极端情况中提取基本真理：

```
真理记录:
- 本质约束: [发现]
- 边界条件: [发现]
- 核心假设: [发现]
- 改进方向: [发现]
```

## 应用场景

### 系统设计

测试系统在不同规模下的表现：

```
用户规模: 1 → 100 → 1万 → 100万 → 1亿
数据量: 1KB → 1MB → 1GB → 1TB → 1PB
```

**洞察**：
- 哪些设计在小规模有效但大规模失效？
- 真正的瓶颈在哪里？
- 需要什么样的架构支持？

### 产品策略

测试产品策略的适用范围：

```
市场: 小众 → 大众 → 全球
价格: 免费 → 低价 → 高价 → 奢侈
功能: 最小 → 完整 → 过度
```

**洞察**：
- 策略的适用边界是什么？
- 何时需要改变策略？
- 核心价值是什么？

### 团队管理

测试管理模式的适用范围：

```
团队: 1人 → 5人 → 20人 → 100人 → 500人
沟通: 直接 → 层级 → 矩阵 → 去中心化
流程: 无 → 轻量 → 标准 → 重型
```

**洞察**：
- 管理模式何时失效？
- 真正的沟通瓶颈是什么？
- 需要什么样的组织结构？

## 常见尺度维度

### 数量尺度

```
极小: 0, 1, 2
小: 10, 100
中: 1K, 10K
大: 100K, 1M
极大: 10M, 100M, 1B+
```

### 时间尺度

```
极短: 毫秒、秒
短: 分钟、小时
中: 天、周
长: 月、年
极长: 十年、世纪
```

### 复杂度尺度

```
极简: 单一功能
简单: 少数功能
中等: 多功能集成
复杂: 复杂系统
极繁: 超复杂系统
```

## 实践技巧

### 双向推演

同时向两个极端推演：

```
当前 → 极小
当前 → 极大
```

### 组合尺度

同时改变多个维度：

```
用户规模 × 功能数量 × 数据量
```

### 现实验证

在现实世界中寻找极端案例：

```
寻找: 已经处于极端的案例
学习: 他们如何应对
应用: 到当前情况
```

## 避免陷阱

- ❌ 只考虑单一维度
- ❌ 忽视极端情况的现实性
- ❌ 过度抽象失去实用性
- ✅ 考虑多个维度的交互
- ✅ 验证极端情况的可行性
- ✅ 提取可行动的洞察

## 输出格式

```
尺度游戏分析：
维度: [测试维度]
极小情况: [描述]
极大情况: [描述]
系统行为:
  - 极小: [反应]
  - 极大: [反应]
边界发现:
  - 上限: [边界条件]
  - 下限: [边界条件]
本质约束: [根本限制]
改进建议: [具体行动]
```

## 示例

### 案例：数据库性能

```
维度: 数据量

极小情况: 100条记录
- 查询时间: <1ms
- 索引: 不需要
- 缓存: 不需要

极大情况: 1亿条记录
- 查询时间: >10s
- 索引: 必需
- 缓存: 必需
- 分片: 可能需要

边界发现:
- 上限: 单表约1000万条记录
- 下限: 索引收益阈值约1万条

本质约束: 磁盘I/O和内存限制

改进建议:
1. 在1万条时添加索引
2. 在100万条时考虑缓存
3. 在1000万条时分表分库
```
