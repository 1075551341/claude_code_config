# 卡住时分派

## 核心原则

```
识别卡住类型 → 选择合适技术 → 执行解决 → 恢复进展
```

## 卡住类型诊断

### 类型1: 信息不足

**症状**：
- 不知道下一步该做什么
- 缺少关键信息
- 不清楚问题本质

**解决技术**：
- 调用 `deep-research` 深度调研
- 调用 `systematic-debugging` 系统化调试
- 调用 `root-cause-analysis` 根因分析

### 类型2: 思维僵化

**症状**：
- 重复同样的思路
- 无法跳出现有框架
- 陷入思维定式

**解决技术**：
- 调用 `collision-zone-thinking` 碰撞思维
- 调用 `inversion-exercise` 反转练习
- 调用 `brainstorming` 头脑风暴

### 类型3: 复杂度过高

**症状**：
- 问题太复杂无法处理
- 组件太多难以理解
- 依赖关系混乱

**解决技术**：
- 调用 `simplification-cascades` 简化级联
- 调用 `meta-pattern-recognition` 元模式识别
- 调用 `scale-game` 尺度游戏

### 类型4: 技术选择困难

**症状**：
- 多个方案无法选择
- 不确定哪个方案最优
- 担心选错方向

**解决技术**：
- 调用 `inversion-exercise` 反转假设
- 调用 `scale-game` 极端测试
- 调用 `competitive-analysis` 竞品分析

### 类型5: 实施困难

**症状**：
- 知道该做什么但做不到
- 缺少资源或能力
- 外部约束阻碍

**解决技术**：
- 调用 `implementation-planning` 实施计划
- 调用 `executing-plans` 计划执行
- 调用 `subagent-driven-development` 子代理协作

## 分派流程

### 步骤1: 诊断卡住类型

```
当前情况: [描述卡住的状态]
尝试过的方案: [列出已尝试的方法]
卡住时长: [持续多长时间]
```

**诊断问题**：
- 是信息问题还是思维问题？
- 是复杂度问题还是选择问题？
- 是理解问题还是实施问题？

### 步骤2: 选择解决技术

根据诊断结果选择最合适的技术：

```
诊断结果 → 推荐技术 → 理由
```

### 步骤3: 执行解决

调用选定的技能并执行：

```
调用技能: [技能名称]
执行步骤: [具体行动]
预期结果: [期望 outcome]
```

### 步骤4: 验证进展

评估解决效果：

```
是否解除卡住: [是/否]
新进展: [描述]
后续行动: [下一步]
```

## 快速决策树

```
卡住了？
├─ 不知道做什么 → 信息不足 → deep-research
├─ 想不出新思路 → 思维僵化 → collision-zone-thinking
├─ 问题太复杂 → 复杂度高 → simplification-cascades
├─ 无法选择方案 → 选择困难 → inversion-exercise
└─ 知道但做不到 → 实施困难 → implementation-planning
```

## 实践技巧

### 时间盒

为每个解决技术设置时间限制：

```
技术A: 30分钟
技术B: 20分钟
技术C: 15分钟
```

### 组合技术

有时需要组合多个技术：

```
第一步: deep-research（获取信息）
第二步: collision-zone-thinking（产生新思路）
第三步: implementation-planning（制定计划）
```

### 记录卡住模式

建立个人卡住模式库：

```
卡住类型: [类型]
触发条件: [何时发生]
有效解决: [什么技术有效]
无效解决: [什么技术无效]
```

## 避免陷阱

- ❌ 在一个技术上反复尝试
- ❌ 忽视卡住的根本原因
- ❌ 盲目尝试各种技术
- ✅ 先诊断再选择技术
- ✅ 设置时间限制
- ✅ 记录和学习卡住模式

## 输出格式

```
卡住分析：
当前情况: [描述]
卡住类型: [诊断结果]
推荐技术: [技能名称]

执行计划: [具体步骤]
时间限制: [设置时间盒]
验证标准: [如何判断解决有效]
```

## 示例

### 案例：性能优化卡住

```
当前情况: 优化数据库查询性能，尝试了多种索引方案都没有明显效果
卡住时长: 2天
尝试方案: 添加索引、优化查询语句、调整缓存策略

卡住类型: 信息不足（不知道真正的瓶颈在哪里）

推荐技术: systematic-debugging

执行理由: 需要系统化地找出真正的性能瓶颈，而不是盲目优化

执行计划:
1. 使用性能分析工具定位慢查询
2. 分析查询执行计划
3. 识别真正的瓶颈
4. 针对性优化

时间限制: 1小时

验证标准: 找到明确的性能瓶颈位置
```
