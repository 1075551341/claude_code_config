# 接收代码审查反馈

## @Examples

```
审查者: "这个函数应该使用 Promise.all 并行处理"
Claude: 理解反馈 → 验证代码库实际调用 → 发现是串行依赖 → 提供反驳证据

审查者: "建议添加抽象工厂模式"
Claude: grep 统计使用点 → 仅1处 → YAGNI → 技术反驳

审查者: "这段代码有 SQL 注入风险"
Claude: 验证确认 → 立即修复 → 测试通过 → 记录修复
```

## 核心原则

**验证后再实施。技术正确性优于社交舒适度。**

## 响应模式

### 标准流程

```
1. 完整阅读反馈
2. 理解/重述需求
3. 对照代码库验证
4. 评估技术合理性
5. 技术性确认或理性反驳
6. 逐项实施，每项测试
```

## 禁止行为

```markdown
❌ 不要说：
- "你说得对！" / "好观点！" / "谢谢指出！"
- 不验证就盲目实施
- 不清楚时部分实施
```

## 应该做

```markdown
✅ 应该做：
- 重述技术需求
- 问澄清性问题
- 技术上不正确时反驳
- "直接修复并在代码中展示"
```

## 处理审查反馈

### 收到反馈时

```markdown
1. 逐条阅读所有反馈
2. 理解每条反馈的意图
3. 验证反馈的技术正确性
4. 分类处理：
   - 明确正确 → 计划实施
   - 需要澄清 → 提问
   - 技术错误 → 反驳
   - 需要讨论 → 邀请讨论
```

### 响应模板

```markdown
## 反馈处理

### 反馈 #1: [反馈标题]

**原反馈**: [审查者的反馈内容]

**理解**: [你对反馈的理解]

**验证结果**: 
- [ ] 代码中确认存在此问题
- [ ] 相关代码位置: `file.ts:45`

**决定**: 
- [ ] 同意，将实施
- [ ] 需要澄清：[问题]
- [ ] 不同意：[技术理由]

**实施方案**: [具体修改方案]

---
```

### 反驳示例

```markdown
❌ 不好的反驳：
"我觉得这样改不太好。"

✅ 好的反驳：
"关于 `validateEmail` 函数的建议，我检查了代码库：

1. 当前实现在 5 处被调用，都依赖现有的返回格式
2. 建议的改动会破坏这 5 处调用
3. 原有实现已覆盖 RFC 5322 标准

建议保持现有实现，或需要同步修改所有调用点

相关代码：
- src/utils/email.ts:15
- src/services/auth.ts:45, 67, 89, 102
"
```

## 外部审查者处理

**保持怀疑态度。**

### 检查项

```markdown
- 技术上是否正确？
  - 实际运行代码验证
  - 检查相关依赖

- 是否会破坏现有功能？
  - 运行测试套件
  - 手动验证相关功能

- 是否与之前决策冲突？
  - 检查文档记录
  - 回顾历史提交

- 是否涉及架构变更？
  - 与团队讨论
  - 更新设计文档
```

### YAGNI 检查

**在实施审查者要求的"专业"功能之前，grep 代码库确认实际使用。**

```markdown
审查者建议："应该添加抽象层以便将来扩展"

检查步骤：
1. grep -r "当前使用模式" src/
2. 统计实际使用点数量
3. 评估扩展的真实需求
4. 如果只有 1-2 处使用，YAGNI 原则可能适用
```

## 应该反驳的情况

```markdown
1. 破坏现有功能
   - 提供具体的影响分析
   - 展示测试失败证据

2. 审查者缺少上下文
   - 提供相关背景信息
   - 指向相关文档

3. 违反 YAGNI 原则
   - 展示当前实际需求
   - 说明过度设计的成本

4. 技术上不正确
   - 提供正确的技术方案
   - 附上参考文档或示例

5. 与用户的决策冲突
   - 引用用户的原始需求
   - 展示冲突点
```

## 反馈分级处理

| 级别 | 处理方式 | 示例 |
|------|----------|------|
| Critical | 必须立即修复 | 安全漏洞、数据丢失 |
| Important | 继续前修复 | 性能问题、边界条件 |
| Minor | 记录后续处理 | 代码风格、命名 |
| Nit | 可选修复 | 注释措辞 |

## 实施修改

### 逐项实施

```markdown
1. 选择一项反馈
2. 理解修改要求
3. 编写修改代码
4. 运行相关测试
5. 验证修改效果
6. 提交修改
7. 标记该项完成
8. 继续下一项
```

### 修改后验证

```markdown
- [ ] 修改的代码编译通过
- [ ] 相关测试全部通过
- [ ] 手动验证修改效果
- [ ] 没有引入新问题
- [ ] 代码风格一致
```

- [待讨论的问题]

## 关键原则

```markdown
1. 理解后再响应
   - 不清楚就问，不要猜测

2. 验证后再实施
   - 技术正确性是第一位

3. 保持专业态度
   - 对事不对人
   - 技术讨论基于证据

4. 记录决策理由
   - 同意或反驳都要有理由
   - 便于后续追溯
```
