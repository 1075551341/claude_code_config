# 上下文工程

## Iron Law

- 上下文使用率 >70% 必须主动压缩
- 上下文使用率 >85% 必须强制压缩或启动新子Agent
- 禁止被动等待上下文溢出，必须在战略断点主动压缩

## 核心策略

### 1. 上下文腐败治理

**症状识别**：
- 输出重复或遗漏关键信息
- 幻觉增加（编造不存在的API/文件）
- 指令遵循度下降
- 响应质量随对话长度退化

**处置方案**：
| 使用率 | 处置                                   |
| ------ | -------------------------------------- |
| <70%   | 正常执行                               |
| 70-85% | 主动压缩：摘要→保留关键决策→丢弃重复  |
| >85%   | 强制压缩或启动新子Agent（fresh window）|

### 2. Token 优化

| 策略                | 方法                                                |
| ------------------- | --------------------------------------------------- |
| 子Agent模型选择     | 简单子任务用 Haiku，复杂用 Sonnet/Opus              |
| 自动压缩阈值       | `AUTOCOMPACT_PCT_OVERRIDE=75`（提前压缩）           |
| 思考Token限制      | `MAX_THINKING_TOKENS=8000`（复杂任务可上调）        |
| meta-prompting      | 精简注入，避免冗余历史                              |
| 战略断点           | 逻辑完成点主动 `/compact`                           |

### 3. 上下文分层

```
顶层：CLAUDE.md（全局规则，始终加载，~4K tokens）
中层：SPEC.md（规范索引，按需引用，~2K tokens）
底层：skills/ agents/（执行能力，触发时加载，按需）
运行时：当前任务上下文（最宝贵，主动管理）
```

### 4. 战略断点压缩

在以下逻辑完成点主动执行 `/compact`：
- 子目标完成后
- 阶段切换时（设计→实现→验证）
- 长时间读文件后
- 重复信息累积时

## 执行步骤

1. 评估当前上下文使用率
2. 识别腐败症状（如有）
3. 选择处置方案（压缩/新Agent/继续）
4. 执行并验证上下文质量
5. 记录模式到 experiences/patterns/
