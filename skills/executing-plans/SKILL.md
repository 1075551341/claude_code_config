# 执行实施计划

## 核心能力

**分步执行计划、进度跟踪、验证检查点。**

---

## 适用场景

- 执行已批准的计划
- 分步开发任务
- 进度跟踪管理
- 检查点验证

---

## 执行原则

### 核心规则

```
1. 按顺序执行，不跳步
2. 每步完成后验证
3. 验证失败则停止调试
4. 记录完成状态
5. 不跳过验证继续
```

### 执行状态

| 状态 | 说明 |
|------|------|
| pending | 待执行 |
| in_progress | 执行中 |
| completed | 已完成验证 |
| blocked | 阻塞等待 |
| failed | 验证失败 |

---

## 执行流程

### 1. 前置检查

```bash
# 确认前置条件
□ 计划文档存在
□ 依赖已就绪
□ 开发环境正常
□ 当前分支正确
```

### 2. 步骤执行

```markdown
## 执行单步流程

1. 标记步骤为 in_progress
2. 执行具体操作
3. 运行验证命令
4. 验证通过 → 标记 completed
5. 验证失败 → 标记 failed，停止
```

### 3. 验证命令

```bash
# 常用验证
npm test           # 测试
npm run build      # 构建
npm run lint       # 检查
tsc --noEmit       # 类型检查
```

### 4. 进度跟踪

```markdown
## 计划执行进度

### 步骤 1：创建基础结构 ✅
- 文件：src/utils/helper.ts
- 验证：编译通过

### 步骤 2：实现核心功能 🔄
- 文件：src/core/engine.ts
- 验证：测试通过

### 步骤 3：集成测试 ⏳
- 依赖：步骤2完成
```

---

## 问题处理

### 验证失败

```
1. 停止执行
2. 调用 systematic-debugging 分析
3. 修复问题
4. 重新验证
5. 继续执行
```

### 计划需要调整

```
1. 记录调整原因
2. 更新计划文档
3. 重新评估影响
4. 继续执行
```

### 遇到阻塞

```
1. 标记为 blocked
2. 说明阻塞原因
3. 记录解决路径
4. 等待或寻求帮助
```
