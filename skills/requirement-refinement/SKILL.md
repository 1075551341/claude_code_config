# 需求细化与确认

## 核心原则

**先宏观理清方向，再逐层拷问细化，确保完全理解并与用户达成一致，然后才能开始实现。**

---

## 何时调用此技能

满足以下任一条件时，应调用此技能：

1. **用户描述不够明确**：内容模糊、存在歧义、缺少关键信息
2. **任务较为复杂**：涉及多个维度、需要拆解、不能直接实现
3. **用户要求细化**：明确要求"详细说明"、"具体要求"、"进一步说明"
4. **需要确认理解**：用户说"确认理解"、"理解正确吗"、"对吗"
5. **涉及规划方案**：需要设计方案、制定计划、评估可行性
6. **用户提出新想法**：概念、建议、创新想法需要深入理解
7. **问题描述不完整**：bug或故障描述缺少关键信息
8. **需要明确边界**：不清楚包含/不包含的内容、边界情况
9. **用户说"帮我实现"但描述模糊**：需要先澄清再实现
10. **任何稍复杂的提问**：不能直接回答，需要先理解背景和细节

---

## 适用场景

- **任务描述**：用户提出任何任务或工作项
- **需求说明**：功能需求、业务需求、技术需求
- **计划方案**：实施计划、技术方案、设计方案
- **问题描述**：bug 描述、问题报告、故障说明
- **概念说明**：任何概念、想法、建议的阐述
- **模糊表述**：任何不够明确、存在歧义的描述
- **复杂提问**：涉及多个方面、需要深度理解的提问

---

## 三阶段核心流程

```
阶段1：宏观理清方向 → 阶段2：逐层分解框架 → 阶段3：循环拷问细化 → 达成一致
```

**关键点**：

- 阶段1：快速建立整体认知，不陷入细节
- 阶段2：建立层次化框架，从宏观到微观逐层分解
- 阶段3：按层次循环拷问，持续细化直到清晰

---

## 阶段1：宏观理清方向

**目标**：快速建立整体认知，明确内容类型和核心意图，不陷入细节。

```markdown
## 宏观理解

**用户原始描述**：
[记录用户的原始内容]

**内容类型识别**：
[任务/需求/计划/方案/问题/概念/其他]

**核心意图复述**：
[用一句话概括核心目标]

**初步疑问方向**：

- 方向1：...
- 方向2：...
- 方向3：...

**确认**：我的宏观理解正确吗？(等待用户确认)
```

**关键原则**：

- 先抓核心，不纠结细节
- 快速建立认知框架
- 确认方向后再深入

---

## 阶段2：逐层分解框架

**目标**：建立层次化分解框架，从一级维度逐层深入到二级维度，明确拷问路径。

```markdown
## 逐层分解框架

### 根节点：[内容类型]

#### 一级维度1：核心目标/意图（宏观层）

- 核心目标是什么？
- 解决什么问题？
- 预期效果如何？

#### 一级维度2：业务维度（中观层）

**任务类型**：

- 任务范围（必须/可选/不包含）
- 执行条件（技术栈/环境/依赖）
- 时间资源（完成时间/优先级/资源）

**需求类型**：

- 用户场景（目标用户/使用场景/频率）
- 功能范围（必须功能/可选功能/不做）
- 技术约束（技术栈/性能/安全）

**计划/方案类型**：

- 实施步骤（具体步骤/依赖关系/并行）
- 资源需求（所需资源/可用性/分配）
- 风险评估（潜在风险/应对措施/备选）

**问题/故障类型**：

- 问题详情（具体表现/发生场景/频率）
- 环境信息（运行环境/相关配置/变更）
- 影响范围（影响功能/影响用户/严重程度）

**概念/想法类型**：

- 核心概念（核心定义/现有关系/创新点）
- 应用场景（应用场景/解决问题/优势）
- 实现可行性（技术可行/所需资源/风险）

#### 一级维度3：技术维度（微观层）

- 具体技术实现细节
- 数据结构设计
- 接口定义
- 性能优化方案

#### 一级维度4：边界与约束（边界层）

- 明确包含的内容？
- 明确不包含的内容？
- 边界情况如何处理？

**确认**：这个逐层分解框架合理吗？需要调整哪些维度？(等待用户确认)
```

**分解原则**：

- 从宏观到微观，逐层深入
- 先业务后技术，先整体后细节
- 每个一级维度可进一步分解为二级子维度

---

## 阶段3：循环拷问细化

**目标**：逐节点循环拷问，持续细化直到每个维度都清晰明确。

### 拷问循环机制

```markdown
## 拷问循环 - [节点名称]

### 第1轮拷问

**问题**：[针对该节点的第一个问题]
**用户回答**：[记录用户回答]
**我的理解**：[复述理解]
**确认**：我的理解正确吗？(等待用户确认)
**是否需要继续拷问**：是/否

### 第2轮拷问（如需要）

**问题**：[基于上一轮回答的追问]
**用户回答**：[记录用户回答]
**我的理解**：[复述理解]
**确认**：我的理解正确吗？(等待用户确认)
**是否需要继续拷问**：是/否

...（继续循环直到该节点清晰）

### 节点完成确认

- [ ] 该节点内容已明确
- [ ] 与用户达成一致
- [ ] 记录最终结论
```

### 拷问顺序（按层次逐层深入）

**第1层：宏观层拷问**

- 拷问核心目标/意图节点
- 确认整体方向和价值
- 明确成功标准

**第2层：中观层拷问**

- 根据内容类型拷问业务维度
- 任务类型：任务范围→执行条件→时间资源
- 需求类型：用户场景→功能范围→技术约束
- 计划方案：实施步骤→资源需求→风险评估
- 问题故障：问题详情→环境信息→影响范围
- 概念想法：核心概念→应用场景→实现可行性

**第3层：微观层拷问**

- 拷问具体技术实现细节
- 数据结构设计
- 接口定义
- 性能优化方案

**第4层：边界层拷问**

- 明确包含/不包含的内容
- 边界情况处理方式

**每个节点的拷问深度**：

- 逐层深入，不跳跃
- 基于回答追问，不预设假设
- 遇到新疑问点，立即澄清
- 每个节点达成一致后再进入下一节点

### 拷问问题清单

**核心目标/意图**：

- 核心目标是什么？
- 解决什么问题？
- 预期效果如何？
- 成功标准是什么？

**任务类型**：

- 必须完成的任务？可选任务？明确不包含的？
- 技术栈/环境/依赖要求？
- 预期完成时间？优先级？可用资源？

**需求类型**：

- 目标用户？使用场景？使用频率？
- 必须功能？可选功能？明确不做的？
- 技术栈/性能/安全要求？

**计划/方案类型**：

- 具体步骤？依赖关系？可并行部分？
- 需要什么资源？资源可用性？
- 潜在风险？应对措施？备选方案？

**问题/故障类型**：

- 问题具体表现？发生场景？频率？
- 运行环境？相关配置？最近变更？
- 影响范围？严重程度？是否紧急？

**概念/想法类型**：

- 核心概念是什么？与现有概念关系？
- 应用场景？解决什么问题？
- 技术可行性？需要什么资源？

**边界与约束**：

- 明确包含的内容？
- 明确不包含的内容？
- 边界情况如何处理？

## 拷问原则

### 核心原则

- **一次一个问题**：避免同时问多个问题，逐个提问
- **基于回答追问**：根据用户回答进行深入追问，不预设
- **避免假设**：不假设用户意图，确认后再继续
- **提供选项**：当需求不明确时，提供选项供用户选择

### 示例

❌ 错误：同时问多个问题

```
"技术栈是什么？性能要求如何？有没有安全要求？"
```

✅ 正确：逐个提问

```
"技术栈有什么要求吗？"
"性能方面有什么要求？"
"安全性方面有什么考虑？"
```

---

## 达成一致的标志

满足以下条件时，认为理解已达成一致：

- 所有树节点都已拷问完成
- 每个节点都有明确的结论
- 用户确认理解正确
- 没有遗留的疑问点
- 边界情况已明确

## 特殊情况处理

### 内容变更

记录变更原因，评估影响，必要时重新拷问相关节点。

### 内容冲突

明确冲突点，分析原因，提供解决方案选项，请用户选择。

### 内容不明确

列出可能的理解，建议先明确核心目标，后续迭代完善。

### 内容超出范围

分解为多个子任务，优先处理核心部分，延后次要部分。

## 输出格式

每次拷问后：

```markdown
## 拷问 - [节点名称]

**问题**：[问题内容]
**用户回答**：[用户回答]
**我的理解**：[复述理解]
**确认**：我的理解正确吗？(等待用户确认)
```

---

## 相关技能

- `design-brainstorming` - 理解确认后的设计头脑风暴
- `implementation-planning` - 基于确认理解的实施计划
- `brainstorming` - 方案讨论和验证
- `systematic-debugging` - 问题故障的系统化调试

---

## 注意事项

- 耐心拷问，不急于得出结论
- 完整记录拷问过程和结论
- 每个问题后确认理解是否正确
- 根据用户反馈灵活调整拷问策略
- 只有达成一致后才进入下一阶段
- 准确识别内容类型，选择合适维度
- 采用树状结构分解，覆盖所有重要维度
