## 分析方法

### 5Why分析法

```markdown
方法：连续追问"为什么"，直到找到根本原因

示例：服务器宕机问题
1. 为什么服务器宕机？
   → 内存耗尽导致OOM

2. 为什么内存耗尽？
   → 某个进程占用大量内存

3. 为什么进程占用大量内存？
   → 存在内存泄漏，对象未释放

4. 为什么存在内存泄漏？
   → 缓存数据未设置过期时间

5. 为什么未设置过期时间？
   → 开发时遗漏，缺少代码审查

根本原因：缓存管理规范缺失，代码审查流程不完善
```

### 鱼骨图分析

```markdown
问题（鱼头）← 原因分类（鱼骨）

分类维度（6M）：
- 人（Man）：人员技能、培训、责任心
- 机（Machine）：设备、工具、基础设施
- 料（Material）：原材料、数据、配置
- 法（Method）：流程、规范、标准
- 环（Environment）：环境、网络、依赖
- 测（Measurement）：监控、指标、阈值

示例：API响应慢问题
                    人                    机
                    │                      │
        代码质量差 ─┼─ 性能意识不足        ├─ 服务器配置低
                    │                      │
        缺少索引 ───┼──────────────────────┼─ 慢查询
                    │                      │
问题：API响应慢 ←──┼──────────────────────┼─ 数据量大
                    │                      │
        无缓存 ─────┼─ 架构设计问题        ├─ 依赖服务慢
                    │                      │
        法                    环
```

### 故障树分析

```markdown
顶事件：系统不可用
│
├── 或门
│   ├── 数据库故障
│   │   ├── 与门
│   │   │   ├── 主库宕机
│   │   │   └── 从库切换失败
│   │   │       └── 原因：配置错误
│   │   └── 连接池耗尽
│   │       └── 原因：未设置最大连接数
│   │
│   └── 应用服务故障
│       ├── 内存溢出
│       │   └── 原因：内存泄漏
│       └── CPU过载
│           └── 原因：死循环
```

## 分析流程

### 标准流程

```markdown
1. 问题确认
   - 明确问题现象
   - 收集问题上下文
   - 确定影响范围
   - 记录发生时间线

2. 数据收集
   - 日志收集
   - 监控数据
   - 配置快照
   - 用户反馈

3. 问题定位
   - 排除法
   - 二分法
   - 对比法
   - 回溯法

4. 根因分析
   - 应用分析方法
   - 识别直接原因
   - 追溯根本原因
   - 确认因果关系

5. 验证确认
   - 复现问题
   - 验证根因
   - 确认逻辑链

6. 改进措施
   - 临时措施
   - 根本措施
   - 预防措施
```

## 常见根因模式

### 代码层面

```markdown
内存问题：
- 内存泄漏（对象未释放）
- 内存溢出（数据量过大）
- 频繁GC（对象创建过多）

性能问题：
- N+1查询
- 缺少索引
- 死循环
- 锁竞争

并发问题：
- 竞态条件
- 死锁
- 资源竞争
- 线程安全
```

### 架构层面

```markdown
设计问题：
- 单点故障
- 无限流熔断
- 无降级策略
- 紧耦合

容量问题：
- 资源预估不足
- 扩容不及时
- 配置不合理

依赖问题：
- 第三方服务不稳定
- 无超时重试机制
- 无熔断保护
```

### 运维层面

```markdown

配置问题：
- 参数配置错误
- 环境不一致
- 配置漂移

流程问题：
- 缺少变更审批
- 测试覆盖不足
- 回滚方案缺失

监控问题：
- 监控盲区
- 告警阈值不当
- 响应不及时
```

### 日志分析

grep -E "ERROR|Exception|Fail" app.log | grep "10:0[0-9]:"

### 代码分析

```python
def analyze_stack_trace(stack_trace):
    """
    分析堆栈信息

    参数：
        stack_trace: 异常堆栈字符串

    返回：
        dict: 分析结果
    """
    import re

    # 提取异常类型
    exception_match = re.search(r'^(\w+Exception|\w+Error)', stack_trace, re.MULTILINE)
    exception_type = exception_match.group(1) if exception_match else 'Unknown'

    # 提取关键堆栈
    stack_lines = re.findall(r'at (.+)\((.+):(\d+)\)', stack_trace)

    # 找到项目代码
    project_code = [
        line for line in stack_lines
        if not any(skip in line[1] for skip in ['site-packages', 'node_modules'])
    ]

    return {
        'exception_type': exception_type,
        'full_stack': stack_lines,
        'project_code': project_code,
        'root_frame': project_code[0] if project_code else None
    }
```
