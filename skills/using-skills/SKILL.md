# 技能系统使用指南

## 核心原则

**相关技能必须在任何响应或行动之前被调用——即使只有 1% 的可能性适用。**

## 触发机制

技能会根据上下文自动触发，触发条件包括：

```markdown
1. 关键词匹配
   - 用户消息中出现触发词
   - 任务描述匹配技能用途

2. 上下文识别
   - 文件类型（.ts, .py, .go 等）
   - 项目结构（package.json, requirements.txt 等）
   - 操作类型（调试、重构、测试等）

3. 显式调用
   - 用户直接请求使用某技能
   - 通过 /skill-name 命令调用
```

## 优先级规则

```markdown
## 优先级顺序

1. 用户指令 > 技能规则
   - 用户明确要求优先执行，则跳过技能检查

2. 流程技能 > 实现技能
   - 先使用 brainstorming 规划
   - 先使用 systematic-debugging 分析
   - 再使用具体实现技能

3. 专用技能 > 通用技能
   - react-component 优于 frontend-developer
   - python-backend 优于 backend-developer
```

## 常见误解

```markdown
❌ 错误想法：
- "这只是个简单问题，不需要技能"
- "技能对这任务太重了"
- "我记得怎么做，不需要参考"
- "用户没明确要求用技能"

✅ 正确做法：
- 即使 1% 可能性也要检查技能
- 简单任务也可能有最佳实践
- 技能提供标准化流程
- 主动检查技能匹配
```

## 技能检查流程

```markdown
## 每次任务开始前

1. 扫描技能列表
   - 哪些技能可能与当前任务相关？

2. 评估匹配度
   - 触发词是否匹配？
   - 任务类型是否吻合？
   - 文件/项目上下文是否相关？

3. 决定是否调用
   - 匹配度 > 1%：调用技能
   - 多个技能匹配：选择最专用的
   - 不确定：向用户确认

4. 执行技能内容
   - 阅读技能完整内容
   - 按技能指导执行
```

## 技能类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 技术 | 具体技术实现方法 | react-component, python-backend |
| 流程 | 工作流程指导 | brainstorming, systematic-debugging |
| 参考 | API 文档和速查 | code-snippets, command-reference |

## 技能调用示例

```markdown
用户："帮我调试这个报错"

分析：
- 任务类型：调试
- 匹配技能：systematic-debugging

行动：
1. 调用 systematic-debugging 技能
2. 按技能流程：观察 → 定位 → 分析 → 修复
```

```markdown
用户："写一个用户登录组件"

分析：
- 任务类型：前端组件开发
- 匹配技能：react-component（如果是 React 项目）

行动：
1. 调用 react-component 技能
2. 按技能模板创建组件
```

## 技能不触发时的检查

```markdown
1. 确认触发词
   - 检查技能描述中的触发词
   - 用户表达是否使用了同义词

2. 检查上下文
   - 项目类型是否匹配
   - 文件类型是否相关

3. 手动调用
   - 使用 /skill-name 命令
   - 明确告诉 AI 使用哪个技能

4. 检查技能是否存在
   - 确认技能已安装
   - 确认技能目录结构正确
```

## 最佳实践

```markdown
1. 不要跳过技能检查
   - 即使任务看起来简单
   - 即使你"记得"怎么做

2. 先流程后实现
   - 先用 brainstorming 规划
   - 再用具体技术技能实现

3. 组合使用技能
   - 设计 + 实现 + 测试
   - 规划 + 执行 + 验证

4. 反馈技能问题
   - 技能内容不准确时更新
   - 缺失场景时补充
```

## 技能更新

```markdown
技能是活的文档：
- 根据实际使用反馈更新
- 发现更好的做法时补充
- 过时内容及时删除
- 触发词根据实际调整
```
