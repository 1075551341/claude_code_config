# 系统化调试

## @Examples

```
用户: "程序崩溃了，错误是 TypeError: Cannot read property 'x' of undefined"
Claude: /systematic-debugging → 收集症状 → 分析现场 → 5Why 根因分析 → 找到 undefined 的来源

用户: "测试全部失败了"
Claude: /systematic-debugging → 分类失败 → 从第一个开始 → 复现 → 定位根因 → 修复
```

## 核心原则

**铁律：未找到根本原因前，绝不尝试任何修复。症状修复是失败。**

```
无根因分析 → 无修复方案
```

---

## 适用场景

适用于任何技术问题：
- 测试失败
- 生产环境 Bug
- 异常行为
- 性能问题
- 构建失败
- 集成问题

**尤其适用于：**
- 时间紧迫时（紧急让猜测变得诱人）
- "只有一个快速修复"看起来明显时
- 已尝试多个修复都失败时
- 上一次修复没生效时
- 未完全理解问题时

## 四阶段流程

### 第一阶段：根因调查

**在任何修复尝试之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **稳定复现**
   - 能可靠触发吗？
   - 精确步骤是什么？
   - 每次都发生吗？
   - 无法复现 → 收集更多数据，不猜测

3. **检查最近变更**
   - 什么变化可能导致此问题？
   - Git diff、最近提交
   - 新依赖、配置变更
   - 环境差异

4. **多组件系统收集证据**
   ```
   对于每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传播
     - 检查每层状态
   ```

5. **追踪数据流**
   - 坏值从哪里产生？
   - 什么调用导致坏值？
   - 向上追踪直到找到源头

### 第二阶段：模式分析

1. **找到工作示例**
   - 在同一代码库中找到类似工作的代码

2. **对比参考实现**
   - 如果实现模式，完整阅读参考实现

3. **识别差异**
   - 工作的和损坏的有什么不同？

4. **理解依赖**
   - 需要哪些其他组件？
   - 做了什么假设？

### 第三阶段：假设与测试

1. **形成单一假设**
   - 明确陈述："我认为 X 是根本原因，因为 Y"

2. **最小化测试**
   - 做最小改变测试假设
   - 一次一个变量

3. **验证后继续**
   - 生效 → 第四阶段
   - 未生效 → 形成新假设
   - 不要叠加更多修复

4. **不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 寻求帮助

### 第四阶段：实现

1. **创建失败测试用例**
   - 最简单的复现
   - 尽可能自动化

2. **实现单一修复**
   - 解决已识别的根本原因
   - 一次一个改变
   - 不捆绑重构

3. **验证修复**
   - 测试通过了？
   - 其他测试没损坏？
   - 问题实际解决了？

4. **如果修复无效**
   - 停止
   - 计数：尝试了多少次修复？
   - < 3：返回第一阶段，用新信息重新分析
   - ≥ 3：停止并质疑架构（见下方）

5. **如果3次以上修复失败：质疑架构**
   - 每次修复在不同地方揭示新问题
   - 修复需要"大规模重构"
   - 每次修复在其他地方创建新症状

   **停止并质疑基础：**
   - 这个模式根本上合理吗？
   - 是否只是惯性坚持？
   - 应该重构架构还是继续修复症状？

## 红色警报 - 停止并遵循流程

如果发现自己思考：
- "现在快速修复，稍后调查"
- "只是尝试改变 X 看看是否工作"
- "同时添加多个改变，运行测试"
- "跳过测试，我手动验证"
- "可能 X，让我修复那个"
- "我不完全理解但这可能工作"
- "模式说 X 但我会不同地适应"
- "这是主要问题：[列出没有调查的修复]"
- 在追踪数据流之前提出解决方案
- "再尝试一次修复"（已尝试2次以上）

**所有这些都意味着：停止。返回第一阶段。**

## 常见借口对照表

| 借口 | 现实 |
|------|------|
| "问题简单，不需要流程" | 简单问题也有根本原因。流程对简单 Bug 很快。 |
| "紧急，没时间走流程" | 系统化调试比猜测检查更快。 |
| "先尝试这个，然后调查" | 第一次修复设定模式。从一开始就做对。 |
| "确认修复工作后写测试" | 未测试的修复不持久。先测试证明它。 |
| "一次多个修复省时间" | 无法隔离什么工作。导致新 Bug。 |
| "参考太长，我会适应模式" | 部分理解保证 Bug。完整阅读它。 |
| "我看到问题，让我修复" | 看到症状 ≠ 理解根本原因。 |

## 速查表

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| **1. 根因** | 读错误、复现、检查变更、收集证据 | 理解什么和为什么 |
| **2. 模式** | 找工作示例、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小测试 | 确认或新假设 |
| **4. 实现** | 创建测试、修复、验证 | Bug 解决，测试通过 |
