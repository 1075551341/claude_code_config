# 并行 Agent 调度

## 核心能力

**同时启动多个子Agent处理独立任务，最大化效率。**

---

## 适用场景

- 批量处理相似任务（如审查多个文件）
- 独立模块并行开发
- 多个独立研究任务
- 代码库广泛搜索
- 多维度分析任务

---

## 调度原则

### 任务可并行条件

```
✅ 可并行：
- 任务间无依赖关系
- 不共享可变状态
- 结果可独立验证
- 失败不影响其他任务

❌ 不可并行：
- 任务间有先后顺序
- 修改同一文件
- 共享资源竞争
- 需要中间结果
```

### 并行收益评估

| 任务数 | 预估时间 | 并行收益 |
|--------|----------|----------|
| 2-3 个 | 各 5min+ | 中等 |
| 4-6 个 | 各 3min+ | 显著 |
| 7+ 个 | 各 2min+ | 高 |

---

## 调度流程

### 1. 任务分解

```markdown
## 任务清单

| ID | 任务描述 | 类型 | 预估时间 |
|----|----------|------|----------|
| T1 | 审查 auth 模块 | review | 5min |
| T2 | 审查 payment 模块 | review | 5min |
| T3 | 审查 user 模块 | review | 5min |

依赖关系：无
可并行：是
```

### 2. Agent 分配

```yaml
agents:
  - type: code-reviewer
    tasks: [T1]
    focus: auth模块安全性
  - type: code-reviewer
    tasks: [T2]
    focus: payment模块事务处理
  - type: code-reviewer
    tasks: [T3]
    focus: user模块数据验证
```

### 3. 执行调度

```markdown
## 并行执行

启动 3 个 code-reviewer Agent：
- Agent 1: 审查 auth/
- Agent 2: 审查 payment/
- Agent 3: 审查 user/

等待全部完成后汇总结果。
```

### 4. 结果汇总

```markdown
## 汇总报告

### Agent 1 结果
- 发现问题：3 个
- 建议修复：...

### Agent 2 结果
- 发现问题：2 个
- 建议修复：...

### Agent 3 结果
- 发现问题：4 个
- 建议修复：...

### 优先级排序
1. [高] payment 模块事务问题
2. [高] auth 模块安全漏洞
3. [中] user 模块验证缺失
```

---

## 最佳实践

### 任务大小控制

```
理想粒度：
- 单个 Agent 处理时间：2-10 分钟
- 结果输出：200-500 行
- 任务边界清晰

避免：
- 过小任务（< 1分钟）→ 调度开销大
- 过大任务（> 15分钟）→ 失去并行优势
```

### 结果合并策略

```markdown
相同类型结果：
- 合并同类问题
- 去重建议
- 按优先级排序

不同类型结果：
- 分类展示
- 标注来源 Agent
- 保留完整上下文
```

### 错误处理

```
单个 Agent 失败：
1. 记录失败任务
2. 继续其他任务
3. 失败任务单独重试
4. 最终报告包含失败信息

多个 Agent 失败：
1. 评估失败影响
2. 决定是否继续
3. 必要时回滚已执行操作
```

## 常用模式

### 批量代码审查

```
启动 N 个 code-reviewer Agent：
- 每个审查一个模块
- 统一审查标准
- 结果汇总排序
```

### 多维度分析

```
启动不同类型 Agent：
- security-scanner：安全角度
- performance-analyzer：性能角度
- code-quality-checker：代码质量角度

综合输出分析报告
```

### 广泛搜索

```
启动多个 Explore Agent：
- 不同目录范围
- 不同关键词
- 合并搜索结果
```
