# 设计头脑风暴

## 核心原则

**在任何实现行动之前，必须完成设计并获得用户批准。**

---

## 适用场景

- 新项目开发
- 功能添加
- 组件构建
- 模块设计
- 架构调整
- 技术方案制定

---

## 禁止清单

**在获得设计批准前不得：**
- 调用任何实现技能
- 编写任何代码
- 搭建任何项目
- 执行任何实现行动

**包括"看起来简单"的项目。**

---

## 流程步骤

### 1. 探索项目上下文

```
检查内容：
- 现有文件结构
- 文档说明
- 最近提交记录
- 当前架构
```

### 2. 评估范围

```
如果请求描述多个独立子系统：
  → 立即标记
  → 帮助用户分解为子项目
  → 每个子项目单独走设计流程

分解原则：
- 识别独立部分
- 确定关系和依赖
- 决定构建顺序
```

### 3. 提问澄清

```
原则：
- 一个问题一次
- 优先选择题
- 开放式也可行
- 理解：目的、约束、成功标准

问题类型：
- 这个功能的目的是什么？
- 有什么约束条件？
- 成功的标准是什么？
- 用户是谁？
- 性能要求？
```

### 4. 提出2-3个方案

```
每个方案包含：
- 技术选型
- 优缺点对比
- 推荐理由
- 实现复杂度

先展示推荐方案并解释原因。
```

### 5. 展示设计

```
设计内容：
- 架构概览
- 组件划分
- 数据流
- 错误处理
- 测试策略

每个部分展示后问"这看起来正确吗？"
根据复杂度调整描述长度：
- 简单：几句话
- 复杂：200-300字
```

### 6. 设计文档

```
保存到：docs/specs/YYYY-MM-DD-<topic>-design.md

检查清单：
- 无占位符（TBD、TODO）
- 内部一致
- 范围聚焦
- 无歧义
```

### 7. 用户审查

```
询问用户：
"设计文档已保存到 <path>，请审查后在实现前告知是否需要修改。"

等待用户确认。
```

### 8. 转向实现

```
用户批准后：
- 调用 implementation-planning 技能
- 创建详细实施计划
- 不调用其他实现技能
```

## 设计原则

### 隔离与清晰

```
每个单元：
- 一个清晰目的
- 定义良好的接口
- 独立理解和测试

边界问题：
- 能否不看内部理解单元？
- 能否改变内部而不破坏消费者？
```

### YAGNI 原则

```
删除所有不必要功能。
保留核心需求。
避免过度设计。
```

### 遵循现有模式

```
在现有代码库工作：
- 先探索当前结构
- 遵循已有模式
- 发现影响工作的问题时，针对性改进
- 不提出无关重构
```

---

## 视觉辅助（可选）

当问题涉及视觉内容时：

```
询问用户：
"有些内容用浏览器展示可能更容易理解。
我可以准备原型、图表、对比等视觉内容。
这个功能较新且可能消耗较多 Token。
想尝试吗？（需要打开本地 URL）"

注意：
- 必须单独发送此询问
- 不与其他内容混合
```

---

## 反模式

```
"这太简单不需要设计" → 每个项目都走流程
"我理解需求，直接实现" → 未澄清假设
"边做边设计" → 先设计后实现
"参考类似项目" → 每个项目独特需求
```
